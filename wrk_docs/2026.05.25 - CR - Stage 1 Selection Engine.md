### SHOULD-FIX

**SF-1 — `select_survivors` will panic on a zero-element group (latent)**

`src/main.rs:173` — `victims.remove(0)` is reached only when `survivors.is_empty()`, which
means `unprotected` holds all the entries. If `group` is empty, `unprotected` is empty and
`remove(0)` panics at runtime.

The two current call sites guard with `files.len() <= 1` / `filter len > 1` before calling
`select_survivors`, so **this is not a live bug today**. But `select_survivors` is documented as
a reusable "single source of truth" and has no contract that prevents callers in later stages
from passing a singleton or empty slice. A future call site (e.g. the read-only metric path in
Stage 2) that iterates over un-filtered groups would trigger it silently.

Fix: add an early return or a `debug_assert!(group.len() >= 2)` with a documented precondition
in the doc comment. Either approach closes the latent panic for all future callers.

**SF-2 — `overall_total` in `process_duplicates` is already stale under non-neutral policy**

`src/main.rs:1215–1219` — `overall_total` is still computed as `Σ (len − 1)` per group before
`select_survivors` is called. When a policy with protect rules is active, the actual victim
count per group may be smaller (or zero for all-protected groups), so `total_candidates` shown
in the "Processed X of Y" status line and in the JSON output would be overstated.

Stage 1 ships with `KeepPolicy::default()` (neutral), so the two values are identical right now
— no user-visible regression exists. The two-metric accounting rework is explicitly deferred to
Stage 2 (HLD §3.5). However, the old formula is inside `process_duplicates`, which now accepts
`&KeepPolicy`, making it easy for a future stage to accidentally pass a non-neutral policy and
get wrong counts with no compile-time warning.

Fix: add a comment at the `overall_total` computation spelling out that this is the pre-protect
formula and must be updated in Stage 2. Optionally `debug_assert_eq!` that `overall_total`
equals the sum of victims under neutral policy (testable because neutral = Lexical + empty
globs).

**SF-3 — Read-only display sort uses `skip(1)` on the raw (unsorted) group**

`src/main.rs:1410` — The groups-for-display sort key is
`group.iter().skip(1).map(|(_, size, _)| *size).sum::<u64>()`. This skips the first entry of
the *raw*, unsorted group to approximate "wasted bytes". Under a neutral policy this is close
enough (survivor is typically near the front after `find_duplicates_in_dirs` appends by walk
order). But it is inconsistent with `select_survivors`'s actual selection, so a group where
the raw `[0]` is a large protected file and `[1..]` are small victims would be sorted wrong in
the display list (it would appear to waste more than it does).

This is pre-existing pre-Stage-1 behavior, but Stage 1 chose to leave it unchanged while
introducing `select_survivors`. The correct fix is `victims.iter().map(…).sum()` from the
`select_survivors` result; that can land in Stage 2 alongside two-metric accounting.

Fix: add a TODO comment on line 1410 noting that this should switch to victim-sum once
two-metric accounting lands.

### NIT

**N-1 — `select_survivors` doc comment claims it replaces `survivor_cmp` "at every call site"
but `collapse_group_by_identity` still uses `survivor_cmp` directly**

`src/main.rs:586` — `collapse_group_by_identity` still calls `survivor_cmp` and is explicitly
left protect-unaware until Stage 3 (HLD §5.6). The doc comment on `select_survivors` should
say "at action and display call sites" rather than "every call site" to avoid confusion.

**N-2 — `mtime_of` is called twice per comparison in `fallback_strategy_cmp` (O(N log N) fs calls)**

`src/main.rs:211–212` — Both `key_a` and `key_b` are fetched via `fs::metadata` on every
invocation of the sort comparator. For the typical tiny duplicate group this is negligible (HLD
§5.3 justifies this), but there is no comment acknowledging it. Add a brief note so a future
reader doesn't try to "optimize" it into a pre-key scheme without checking the design rationale.

**N-3 — `Newest` fallback logic is correct but non-obvious; a short comment would help**

`src/main.rs:214` — `cmp_oldest(&key_b, &key_a)` reverses the argument order rather than
calling a separate `cmp_newest`. This is correct (reversal inverts the sort direction), but the
swap is easy to misread. A one-line comment like `// swap args to reverse: largest mtime sorts
first` would prevent a future reader from questioning whether it is a bug.

**N-4 — `any_ancestor_dir_matches` matches against Windows `Prefix` and `RootDir` components**

`src/main.rs:117–122` — On Windows, `path.components()` begins with a `Prefix` component
(`"C:"`) and a `RootDir` component (`"\"`). These are passed to `set.is_match()`. For the
default glob `0*` or any sane user glob these never match, but an unusual user glob like `C*`
would match the drive letter prefix on every Windows path, silently protecting all files. This
is an edge case the HLD does not address.

Fix (low priority): filter to only `Component::Normal` variants before matching, or add a note
in the doc comment that `Prefix`/`RootDir` components are visible to the glob matcher on
Windows.

**N-5 — `test_select_survivors_dir_first_reason_when_both_match` uses a single-element group**

`src/main.rs` (test at line ~6046) — This test exercises the "dir label wins when both rules
match" path with a group of one entry. That is a valid structural test, but it cannot verify
that the victim list is correct (there is nothing to remove). The assertion is `survivors.len()
== 1` with an implicit check that `_victims` is empty (unchecked). Consider adding an explicit
`assert!(victims.is_empty())` for completeness, and optionally add a second entry that matches
only the name rule (so the dir-first label is tested against actual selection, not just a
singleton).

**N-6 — `policy_with` helper is not annotated; it is pub-visible inside the test module**

Minor: `policy_with`, `entry`, `survivor_paths`, `victim_paths` helper functions are defined
without `#[allow(dead_code)]` and are not `#[cfg(test)]`-gated at the function level (they are
inside `mod tests` which is `#[cfg(test)]`, so this is fine). No action needed; calling out for
clarity.

## Neutral-Default Invariant Verification

With `KeepPolicy::default()` (empty `protect_dir`, empty `protect_name`, `FallbackStrategy::Lexical`):

1. `any_ancestor_dir_matches` returns `false` for every path (empty `GlobSet`).
2. `file_name_matches` returns `false` for every path.
3. All entries go to `unprotected`.
4. `unprotected.sort_by(|a, b| fallback_cmp(a, b, Lexical))`.
5. `fallback_cmp` with `Lexical` = `root_a.cmp(root_b).then_with(|| ∅).then_with(|| path_a.cmp(path_b))`
   = `root_a.cmp(root_b).then_with(|| path_a.cmp(path_b))` = **`survivor_cmp` exactly**.
6. `victims.remove(0)` → keeper is the `survivor_cmp`-minimum element.
7. Remaining `victims` = `skip(1)` of the sorted slice.

The invariant holds. There is no path where `KeepPolicy::default()` diverges from the old
`sort_by(survivor_cmp) + skip(1)` behavior, including through: the mtime branch (not reached
under `Lexical`), the protect branch (not reached with empty globs), and the identity-collapse
path (unchanged, still uses `survivor_cmp` directly).

## Fallback Chain Correctness

| Strategy | Sort key | None (unreadable mtime) | Deterministic? |
|---|---|---|---|
| `Lexical` | ∅ (path alone) | N/A | Yes — `path` is total order |
| `Shortest` | `components().count()` ascending | N/A | Yes — integer, then `path` |
| `Oldest` | mtime ascending | `None > Some(_)` → loses (treated newest) | Yes — `path` tiebreak |
| `Newest` | mtime descending (args swapped) | `None > Some(_)` → first after swap → wins (treated newest) | Yes — `path` tiebreak |

The `None`-as-newest semantics are correct per HLD §5.3 and consistently implemented for both
`Oldest` and `Newest` branches. The `path` final tiebreak guarantees a strict total order under
all strategies.
