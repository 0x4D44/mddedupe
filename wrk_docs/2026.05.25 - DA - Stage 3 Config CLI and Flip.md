# Devil's Advocate — Stage 3: Config, CLI Flags & the Default Flip

- **Date:** 2026-05-25
- **Target:** `feature/configurable-survivor-selection` (uncommitted working tree)
- **Stage:** 3 — turn the `0`/`00-` convention ON by default
- **Reviewer stance:** adversarial, read-only. All line refs are to the working
  tree at review time.

Verdict up front: the engine is solid and all 206 tests pass, but **the flip
ships a real, silent survivor-identity change that the test suite does not pin
where it matters**, and **the same group is partitioned independently 4× per run
with on-demand mtime**, so the file shown as KEPT can differ from the file
actually kept. Two BLOCKING issues, three SHOULD-ADDRESS, plus doc drift.

---

## 1. The default flip changes survivor identity silently (attack #1)

**Status: BLOCKING (safety) — partially mitigated, gaps remain.**

The CLI default is now `protect-dir=["0*"]`, `protect-name=["00-*","00 - *"]`,
`fallback=oldest` (`src/main.rs:97-100`, `resolve_keep_policy` 1639-..., wired at
`run_app:1672`). For an **unprotected single-root group** the keeper is now the
**earliest mtime** instead of the lexical/`root_index` winner.

`tests/cli.rs:create_duplicate_fixture()` writes `file1.txt` then `file2.txt`
back-to-back. Under `oldest`, the keeper depends on whether the two writes land
in the same filesystem mtime tick:
- equal mtime → `path` tiebreak → `file1.txt` survives (today's answer, by luck);
- distinct mtime → `file1.txt` is older → still survives.
So the *outcome* happens to coincide here, but **only because the fallback chain
keeps `path` as the final tiebreak and the older file also sorts first
lexically.** That is a coincidence of this fixture, not a guarantee.

What is actually protective: the two identity-asserting tests
(`cli_action_delete_force_removes_duplicate:34`, the variadic test at `:494`)
were correctly pinned with `--no-protect --keep lexical`. Good.

The gap the author should not hand-wave:
- `cli_json_summary_outputs_valid_json` (`tests/cli.rs:112`) runs a **bare**
  scan and its own comment (`:143-150`) asserts the metrics hold "under the
  **neutral default**". It is **not** the neutral default — it is the convention
  default. The assertion passes because `file*.txt` matches no marker, but the
  comment documents the wrong mental model, and any future fixture rename to a
  `0`/`00-` name would flip removable≠redundancy and break it for a reason the
  comment actively misdirects you away from.
- **Disproved sub-claim:** could an `assert_fs`/`tempfile` path component start
  with `0` and silently protect a whole fixture? No. `tempfile` prefixes every
  generated dir with `.tmp`, so the component is `.tmpXXXXXX` (starts with `.`),
  and the `%TEMP%` base (`C:\Users\<user>\AppData\Local\Temp`) has no
  `0`-leading component. `0*` is anchored at component start, so no temp path
  matches. This one is safe — but it is safe by luck of the prefix, not by
  design, and nothing tests it.

**Ask:** add a single-root, equal-content, *distinct-explicit-mtime* integration
test that asserts which file survives under the bare default (HLD test #6/#12
exist as unit-ish coverage, but no *CLI* test pins survivor identity under the
live `oldest` default). Fix the misleading "neutral default" comment at
`cli.rs:143`.

## 2. Displayed survivor can disagree with the acted survivor (attack #5)

**Status: BLOCKING (correctness/safety).**

"Single source of truth" (HLD §5.2/§4.3) is true of the *function*, not of a
*result*. `select_survivors` is called **independently at four sites** in one
run, each re-reading mtime via `mtime_of` (`src/main.rs:134`, no caching):

1. metrics pass — `find_duplicates_in_dirs:1027`;
2. listing sort key — `run_app:1761`;
3. listing display — `run_app:1781`;
4. action pass — `process_duplicates` (diff lines ~196-214).

For `oldest`/`newest` these reads happen at different wall-clock instants. If a
file's mtime changes between the listing pass (3) and the action pass (4) — a
background indexer, antivirus, a sync client, or simply a long scan on a large
tree — the listing can print `KEPT (oldest)` for file A while the action deletes
A and keeps B. For a **destructive** tool that is the exact failure the shared-
function invariant was supposed to prevent. The counts still agree (len-1
victims either way), so the summary will not betray it; only the *identity*
silently drifts.

The mitigating comment ("mtime is read on demand ... assumed stable for the
duration of the scan", diff ~197-202) lives **only** in `process_duplicates`,
is not surfaced to the user, and is a genuinely shaky assumption for a dedup
tool that may run for minutes. The HLD never states "mtime stable during scan"
as a documented precondition (§5.3 only justifies on-demand reads for
performance).

**Ask:** partition each group **once**, cache `(survivors, victims)` (or at
least the mtime keys), and feed the same result to metrics, listing, and action.
Then the invariant is real, not aspirational. If caching is rejected, document
the TOCTOU window in user-facing docs and §5.3, not just a code comment.

Related, non-blocking: with `--action delete` at Info level the group listing is
printed **twice** (read-only block at `1773-1799` *and* `process_duplicates`
when `info_logs`). Pre-existing, but Stage 3 now decorates both with verbs, so
the duplication is louder and reinforces #2 (two separate evaluations the user
can visually compare and find disagreeing).

## 3. `--no-protect` is not "today's behavior" (attack #2)

**Status: SHOULD-ADDRESS (HLD inconsistency, surprising semantics).**

Impl: `--no-protect` clears both protect lists but **leaves fallback at
`oldest`** (`resolve_keep_policy:1645-1657` + doc-comment 1633-1635). So
`--no-protect` alone ≠ pre-Stage-3 behavior; you need `--no-protect --keep
lexical`.

This matches HLD **§4.2** ("`--no-protect` ⇒ both protect lists empty"; fallback
handled separately) but **contradicts HLD §3.3**, which describes the opt-out as
"a neutral policy (no protect, **`lexical` fallback** — i.e. today's exact
behavior) ... reachable via `--no-protect`". The HLD is internally inconsistent
and the impl silently chose the §4.2 reading.

Least-surprising? Debatable. A user typing `--no-protect` to "turn this off and
get the old result" gets a *different survivor* (`oldest` vs historical
`lexical`) with no warning. It is tested (`test_resolve_keep_policy_no_protect_empties_lists`
pins fallback = `Oldest`) and documented in the code comment, so it is at least
deliberate — but the HLD §3.3 wording must be corrected, and the team should
decide whether `--no-protect` should reset fallback to `lexical` for true
"today's behavior" parity.

## 4. cwd-relative `./.mddedupe.toml` auto-load is a latent unit-test flake (attack #3)

**Status: SHOULD-ADDRESS.**

`load_config` (`src/main.rs:1573-...`) reads `./.mddedupe.toml` relative to the
**process cwd** when no `--config` is given. Three unit tests call
`resolve_keep_policy` with `config: None` and do **not** control cwd:
`test_resolve_keep_policy_no_config_is_convention_default`,
`..._no_protect_empties_lists`, `..._bad_glob_is_fatal`. They pass only because
no `.mddedupe.toml` currently exists in the repo root (verified: none present).
A developer dogfooding the tool who drops a `.mddedupe.toml` in the repo root
would see these unit tests **and** the bare-cwd CLI tests
(`cli_bad_protect_glob_exits_non_zero` etc.) silently change behavior or flake.
This is global mutable process state leaking into the suite.

The integration test `cli_default_convention_protects_master_copy` *is* hermetic
(`.current_dir(temp.path())`, a config-less dir) — good, and that pattern should
be applied to the cwd-sensitive unit tests too (set cwd to a temp dir, or always
pass an explicit `--config`/sentinel).

Precedence correctness is right: `--config <missing>` is **fatal**
(`ConfigUnreadable`, tested `cli_bad_config_path_exits_non_zero:818` and unit
`..._missing_config_is_fatal`), while an absent `./.mddedupe.toml` is a **silent
default** (`load_config` returns `Ok(None)` without touching disk). The two
correctly differ. `deny_unknown_fields` genuinely fires: `ConfigFile` and
`KeepConfig` both carry it (`src/main.rs:105,113`) and a typo'd `protect-dirs`
is proven fatal by `test_resolve_keep_policy_unknown_field_rejected` against a
real malformed file. This sub-area is well covered.

## 5. Explain-output honesty (attack #4)

**Status: QUESTION — mostly honest, two caveats.**

- **No "would delete" lies about protected files.** In both the read-only block
  and `process_duplicates`, survivors always get `KEPT (...)` and only victims
  get the verb (`run_app:1782-1797`, diff ~232-260). A protected file is always
  in the survivor list, so it can never be labelled with an action verb.
  Disproved.
- **Both-match labelling is correct but under-specified.** A file matching dir
  AND name reports `protected: dir` (`protect_reason` tests dir first,
  `src/main.rs:163-172`), matching §3.2. But `survivor_reason_label`
  (`:195-205`) reports the **deepest** matching ancestor (`.rev().skip(1)`), so
  `0root/00-sub/x` says `dir 00-sub`, not `dir 0root`. Cosmetic; §3.2/§5.7 don't
  pin which ancestor name to show. Note only.
- **Verbosity gating holds for the summary**, but the *listing* leaks in JSON
  mode. `KEPT`/verb lines print whenever `log_output` is true
  (`run_app:1675,1751`), which is independent of `summary_format`. In
  `--summary-format json` **without** `--quiet` at Info level, the decorated
  human listing prints to stdout *before* the JSON object. This is pre-existing
  (the JSON test uses `--quiet`, and parsing does `find('{')`), but Stage 3
  makes the leaked preamble noisier. `--quiet`/`--summary-only` correctly
  suppress it (`log_output` is false). Worth a doc note that JSON consumers must
  pass `--quiet`.

## 6. Glob breadth of `0*` (attack #6)

**Status: QUESTION — acknowledged risk, no new over-match found.**

`any_ancestor_dir_matches` (`src/main.rs:141-149`) matches each component string
(not a path fragment), case-insensitive, anchored at component start. Confirmed:
- Windows `Prefix` component (`C:`) and `RootDir` do not match `0*` — no
  drive-root over-match.
- `0*` matches files vs dirs correctly: name rules use `file_name()`, dir rules
  skip the final component (`take(dir_count)`), so a *file* literally named `0x`
  is not dir-protected by `0*` (only name globs apply to it). Correct.
- The genuine breadth risk is the one HLD §10 already owns: a top-level dir
  literally named `0`, `0_inbox`, `0day`, `007`, `2020`→no (needs leading `0`),
  etc. silently protects its entire subtree from a bare `--action delete`. This
  is the headline behavioural risk of the flip and is exactly why #1's missing
  CLI survivor-identity test matters.

No surprising component-vs-fragment confusion: globset never sees a separator
because matching is per-component (§5.7 honoured).

## 7. Could the new tests pass with a broken merge/precedence impl? (attack #7)

**Status: SHOULD-ADDRESS — coverage is good but has one blind spot.**

The `resolve_keep_policy` unit tests are reasonably adversarial:
`flag_overrides_config` proves flag *replaces* (not merges) the config dir list
AND asserts the untouched `protect-name` falls through to the **default** (not
empty), which would catch a "clear everything on any flag" bug;
`omitted_config_field_uses_default` and `explicit_empty_list_disables`
distinguish `None` from `Some(vec![])`, which is the whole point of
`Option<Vec<_>>`. A naive impl that ignored the `Option` distinction would fail
these. Good.

Blind spots:
- **No test exercises config + `--protect-name` flag together** where config
  *also* sets `protect-name`, to prove the flag wins specifically on the name
  list (only the *dir* override-vs-config interaction is tested at
  `flag_overrides_config`). A bug that applied flag-replace to dir but
  config-precedence to name would survive.
- **No test combines `./.mddedupe.toml` auto-discovery with a flag** — every
  config test uses explicit `--config`. The cwd-discovery branch
  (`load_config:1576-1583`) is only exercised incidentally and never asserted to
  load. A broken auto-discovery path (e.g. wrong filename, swallowed error)
  would not be caught by any test.
- The two-metric divergence test (`cli_divergent_two_metric_summary`) only
  checks substrings (`"2 duplicate files"`, `"1 removable"`, `"1 protected and
  kept"`); it does not assert *which* file was kept, so it would pass even if
  the wrong protected copy were chosen as the representative — but since all
  protected copies survive, that is acceptable here.
