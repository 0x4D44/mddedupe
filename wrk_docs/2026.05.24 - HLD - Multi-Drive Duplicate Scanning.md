# HLD тАФ Multi-Drive Duplicate Scanning

- **Date:** 2026-05-24
- **Component:** `mddedupe`
- **Status:** Proposed (rev. 2 тАФ scan-origin priority + aliasing safety)
- **Author:** Arthur + Claude

## 1. Summary

Extend `mddedupe` to scan **two or more paths** in a single run and find
duplicates **across** them, e.g.:

```
mddedupe X:\ Y:\
```

When a set of identical files spans the supplied paths, the tool keeps the
copy under the **earliest-listed** path and consolidates the rest. A single
path behaves exactly as it does today, so the change is backwards compatible
at the CLI level.

## 2. Goals / Non-Goals

### Goals
- Accept 1..N positional paths instead of exactly one.
- Find duplicates across all supplied paths (not just within each one).
- Prefer to keep the survivor on the earliest-listed path ("first drive wins").
- Preserve all existing behavior for the single-path case.
- Never delete the last physical copy of a file, even when it is reachable
  through more than one of the supplied paths (see ┬з5.4).

### Non-Goals
- No migration, rollback, or backwards-compat shims (development sandbox).
- No new environment variables or feature flags.
- No per-file identity check in the **hot scan loop**. Identity is only resolved
  *within* already-filtered duplicate groups, which are tiny (see ┬з5.4).
- No change to the move/trash/delete actions themselves.
- No cross-root symlink reconciliation; `--follow-symlinks` across multiple roots
  is documented as unsupported (see ┬з5.5).

## 3. Behavior

### 3.1 Consolidation model ("Consolidate, prefer earliest path")

Each set of byte-identical files (one SHA-256 hash group) collapses to exactly
**one** survivor тАФ the same "one survivor per group" rule the tool already uses.
The only change is **how the survivor is chosen**:

> Sort each group by `(root_index, path)` and keep the first.
> `root_index` is the position of the supplied path the file was discovered
> under (0 = first path listed). Ties within the same path fall back to
> alphabetical order (unchanged from today).

Consequences (all consistent with the consolidate model):
- Redundant copies **within** the first path are still consolidated.
- A set living **entirely** on a later path is still consolidated (survivor on
  that path).
- The action (move/trash/delete) is applied to every non-survivor, as today.

### 3.2 Worked example

Input тАФ `X:\` listed first, `Y:\` second:

| File   | Content |
|--------|---------|
| `X:\a1` | C |
| `X:\a2` | C |
| `Y:\b1` | C |
| `Y:\b2` | D |
| `Y:\b3` | D |

Result:

| Set | Survivor | Acted on |
|-----|----------|----------|
| C `{X:\a1, X:\a2, Y:\b1}` | `X:\a1` | `X:\a2`, `Y:\b1` |
| D `{Y:\b2, Y:\b3}`        | `Y:\b2` | `Y:\b3` |

Final state: `X:\a1`, `Y:\b2`.

### 3.3 Overlapping paths

If one supplied path is inside another (or the same path is given twice), the
tool exits with a clear error **before scanning anything**:

```
$ mddedupe X:\ X:\photos
Error: path 'X:\photos' is inside 'X:\'. Pass non-overlapping paths.
```

Exit code `1`. Detection is done on **canonicalized** paths (┬з5.3), so it also
catches roots that alias each other through symlinks or junctions, not just
lexical nesting.

## 4. CLI / Output Changes

- **Positional arg:** `directory: PathBuf` тЖТ `directories: Vec<PathBuf>`
  (clap `required = true`, `num_args = 1..`).
- **Startup line:** `Starting duplicate scan in directory: X` тЖТ lists all paths.
- **JSON summary:** field `directory: String` тЖТ `directories: [String]`
  (verified: no test or consumer reads the old field).
- **Read-only listing:** files within each duplicate group are printed in
  `(root_index, path)` order so the survivor appears first.
- **Help text:** `about` / `long_about` updated for plural paths.

No flags are added or removed.

## 5. Design Detail

### 5.1 Scanning and origin tracking (`src/main.rs`)

The core scanner changes signature from a single `dir: &Path` to
`dirs: &[PathBuf]` and walks one `WalkDir` per path, recording **which path each
file came from** by its index. This is the key correctness decision: priority is
assigned at the moment of discovery rather than reconstructed later by
path-prefix matching (which is fragile around case, trailing separators, `..`,
and symlinked roots).

The carried tuples gain the origin index (matching the codebase's existing
tuple style):

- `size_map: HashMap<u64, Vec<(PathBuf, usize)>>` тАФ value carries `root_index`.
- duplicates map: `HashMap<String, Vec<(PathBuf, u64, usize)>>`.

The scanned-file counter and progress reporting accumulate across paths
unchanged; the "Filesystem scan complete" line is printed once after all paths.

A single-path convenience helper lives **inside `#[cfg(test)] mod tests`** (not
production) so existing scan tests don't churn:

```rust
// test module only
fn scan_one(dir: &Path, emit_text: bool, show_progress: bool, follow: bool)
    -> io::Result<...> {
    find_duplicates_in_dirs(&[dir.to_path_buf()], emit_text, show_progress, follow)
}
```

### 5.2 Survivor selection

Because the origin index travels with each candidate, survivor selection is a
local sort with **no extra parameters**:

```rust
// in process_duplicates, per group:
group.sort_by(|(pa, _, ra), (pb, _, rb)| ra.cmp(rb).then_with(|| pa.cmp(pb)));
// keep group[0], act on the rest
```

`process_duplicates` no longer needs a `roots` argument. `total_candidates`
(sum of `len-1` per group) is unchanged, since the model still keeps exactly one
survivor per group, and every member of a group shares the same size so wasted
-byte totals are independent of which copy survives.

### 5.3 Overlap detection

Checked once at the top of `run_app`, before the scan:

```rust
fn detect_overlap(dirs: &[PathBuf]) -> Option<(PathBuf, PathBuf)>; // (outer, inner)
```

- Canonicalize each path best-effort (fall back to the raw path if it doesn't
  exist yet). Canonicalization resolves symlinks/junctions, so aliased roots are
  caught.
- Compare canonical forms pairwise with `starts_with`; equal paths count as
  overlap.
- On a hit, `run_app` returns `AppError::OverlappingPaths(outer, inner)`
  (exit 1, message per ┬з3.3).

### 5.4 Identity safety net (never delete the last copy)

Overlap rejection is lexical/canonical at the *root* level. It does **not** stop
the same physical file from entering a hash group twice via a symlink or
junction discovered mid-walk. Acting on such a "duplicate" would destroy the
data the survivor itself points at.

Guard: **after hashing, collapse each hash group by `FileId`** so a single
physical file appears at most once before survivor selection, reporting, or
actions. When collapsing entries that share a `FileId`, keep the **lowest**
`root_index` so the survivor still prefers the earliest path.

- `FileId` already exists (`dev`+`ino` on Unix, canonical path elsewhere) and is
  computed only for members of already-filtered duplicate groups тАФ a tiny set,
  so there is no hot-loop cost.
- This also fixes a **pre-existing** latent bug: single-path `--follow-symlinks`
  can today pair a real file with a symlinked view of itself and delete the
  target. Collapsing by identity closes that too.

**Hardlinks and the Unix/Windows asymmetry (intended).** Because identity is the
`FileId`, on **Unix** two or more hardlinks to the same inode share one
`FileId` (`dev`+`ino`) and collapse to a single entry. They are therefore
**counted once and never unlinked** тАФ which is exactly right for a
space-reclaiming deduper: hardlinks already share one physical extent on disk,
so deleting one link frees nothing, and removing the last link would destroy the
only copy. Collapsing them keeps `wasted_bytes` accurate (a hardlink is not
wasted space) and prevents acting on a "duplicate" that costs no storage. This
behavior is **asymmetric across platforms**: on **Windows** identity is the
canonical path, and each NTFS hardlink canonicalizes to its *own* distinct path,
so hardlinks are **not** collapsed тАФ they are reported and acted on as ordinary
duplicates. We accept this asymmetry rather than engineering Windows hardlink
detection (`FileIndex` via `GetFileInformationByHandle`), which is out of scope.

### 5.5 `--follow-symlinks` with multiple roots

Symlinks that cross between roots create genuinely ambiguous ownership and
visit-ordering. We do **not** attempt cross-root reconciliation. The ┬з5.4
identity collapse guarantees *safety* (no last-copy deletion); correctness of
*counts* in pathological cross-root symlink setups is not guaranteed. This is
documented as unsupported rather than engineered around.

### 5.6 New error

```rust
enum AppError {
    // ...
    OverlappingPaths(PathBuf, PathBuf),
}
```

`format_app_error` maps it to the ┬з3.3 message with exit code `1`.

## 7. Testing

Driven test-first; `cargo test` / `clippy` / `fmt` stay green.

New tests:
1. **Cross-path consolidation** тАФ a duplicate across two temp dirs is removed
   from the second; the survivor stays in the first (the ┬з3.2 shape, scaled down).
2. **First-path priority wins over alphabetical** тАФ given paths ordered `[B, A]`
   where temp-dir names sort `A < B`, the survivor is the copy under `B` (proves
   `root_index` beats path order).
3. **Within-later-path consolidation** тАФ a set existing only under the second
   path still collapses to one survivor.
4. **Overlap rejection** тАФ nested paths return `AppError::OverlappingPaths`;
   nothing on disk is modified.
5. **Identity safety net (unix)** тАФ a file reachable twice (symlink + follow)
   resolves to one entry: it is **not** deleted and does not inflate the
   duplicate count.
6. **Cross-device move** тАФ multi-path scan with `--action move --dest тАж`
   exercises the `relocate_file` EXDEV copy fallback (now the common path).
7. **clap parsing** тАФ variadic positional parses correctly when interleaved with
   options (`dir --action delete`, `--quiet dir1 dir2`).

Existing tests adapt mechanically (field rename, `root_index` in map literals,
`scan_one`) and otherwise assert identical behavior, protecting the single-path
path.

## 8. Risks

- **Aliasing / last-copy deletion:** mitigated by canonical overlap detection
  (┬з5.3) plus identity collapse of hash groups (┬з5.4). This is the primary
  safety property and is covered by test 5.
- **clap variadic positional vs. interleaved options:** must-verify, covered by
  test 7; expected to work but not assumed.
- **Large two-drive scans:** no per-file canonicalization in the scan loop
  (origin index is free; `FileId` only touches duplicate-group members), so
  single-path performance characteristics are preserved.
- **`--follow-symlinks` across roots:** count accuracy not guaranteed; documented
  as unsupported (┬з5.5). Safety still holds via ┬з5.4.
