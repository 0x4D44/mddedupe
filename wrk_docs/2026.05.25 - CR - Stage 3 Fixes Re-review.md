## SHOULD-FIX

### S1. Redundant `CreateDestRequiresMove` guard — the `None` arm is dead code

**Location:** `src/main.rs` lines 1739–1756

There are now two independent guards for the same condition:

1. **Top guard (line 1739–1743):** `if args.create_dest && action_str != "move" {
   return Err(CreateDestRequiresMove) }`. This fires before the `match` is
   evaluated, covering every case where the action string is anything but `"move"`
   — including `None`.

2. **`None` arm inside the match (lines 1753–1756):** also checks `args.create_dest`
   and returns the same error. This arm is unreachable: if `args.action` is `None`
   and `args.create_dest` is `true`, the top guard at line 1739 fires first.

The logic is correct — both paths return the same error — but the `None` arm's
guard is dead code that could mislead a future reader into thinking the top guard
only fires for Some actions. The top guard was pre-existing; the `None` arm was
added during the action-parsing refactor.

**Fix:** Remove the `if args.create_dest { return Err(...) }` block inside the
`None` arm and let it simply return `None`. The top guard already covers it.

## NITs

### N1. New `AppError` variants have no `test_format_app_error_*` unit tests

**Location:** `src/main.rs` — `format_app_error` (approx. line 2081–2092)

Every pre-existing `AppError` variant has a dedicated `test_format_app_error_*`
test verifying the message text and exit code. The three new variants
(`ConfigUnreadable`, `ConfigParse`, `InvalidGlob`) are covered end-to-end by
integration tests but have no unit-level `format_app_error` tests. The error
messages are simple; this is just consistency with the existing pattern.

### N2. `action_str` is bound but unused in the `trash` and `delete` match arms

**Location:** `src/main.rs` lines 1773, 1776

```
Some(action_str) if action_str.eq_ignore_ascii_case("trash") => ...
Some(action_str) if action_str.eq_ignore_ascii_case("delete") => ...
```

`action_str` is bound in the pattern but not referenced in the arm body. Clippy
does not warn (the binding is used in the guard), but `Some(_) if ...` would be
slightly cleaner and make the intent explicit. Not a bug.

### N3. `cli_divergent_two_metric_summary` comment says "under `0*` dirs" but `00-a`/`00-b` match via `0*`

**Location:** `tests/cli.rs` — `cli_divergent_two_metric_summary` (approx.
line 849)

The comment says "Two protected copies (under `0*` dirs)" — this is correct, but
it could be made more precise: the dirs `00-a` and `00-b` match `0*` (dir glob).
The file name `x.txt` does not match `00-*` (name glob), so dir protection is what
fires. A reader might assume `00-*` dirs trigger the name rule; a one-sentence
clarification would prevent confusion.

### N4. `cli_no_protect_lexical_reverts_to_root_index` verifies only the listing, not the filesystem outcome

**Location:** `tests/cli.rs` — `cli_no_protect_lexical_reverts_to_root_index`

The test runs read-only (no `--action`) and verifies the listing labels. It does
not verify the filesystem outcome after a delete action under the neutral policy.
This is acceptable — the invariant that display equals action is tested by the
unit test `test_group_plan_display_action_metrics_agree_under_oldest` and the
integration test `cli_action_delete_force_removes_duplicate`. Worth a comment in
this test noting why filesystem verification is omitted.

## Cross-Cutting Checks

### Single-plan integrity

Confirmed. `build_group_plans` is called once. No independent partition derivation
exists anywhere in the post-fix code. The prior sites in `find_duplicates_in_dirs`
(the old accounting loop calling `select_survivors`) are removed. The prior sites
in the display-sort and display-print loops are replaced by iteration over `plans`.

### JSON metrics correctness

`JsonSummary.removable_files` and `.reclaimable_bytes` are assigned from the local
variables at lines 1991–1992, which were set from `plans` at lines 1826–1827.
`DuplicateScan` no longer exposes these fields. Under the neutral/common case
(one survivor per group), `removable_files == redundant_files` and
`reclaimable_bytes == redundant_bytes`, matching prior behaviour. This invariant
is proved by `test_neutral_parity_removable_equals_redundancy`.

### `process_duplicates(&[GroupPlan])` call sites

All eight call sites in the test module are correctly updated to
`&build_group_plans(&duplicates, &KeepPolicy::default())`. The call in `run_app`
(line 1904) passes `&plans`. Cancellation via `cancellation_requested()` is
checked per-victim inside `process_duplicates` (unchanged). `total_candidates` is
computed as `plans.iter().map(|p| p.victims.len()).sum()`.

### `--no-protect` semantics

- Empty protects: verified by `test_resolve_keep_policy_no_protect_is_neutral_today_parity`.
- Fallback = `Lexical`: verified by same test.
- `--keep` overrides the neutral fallback: verified by `test_resolve_keep_policy_no_protect_keep_overrides_neutral_fallback`.
- Config `fallback` is ignored under `--no-protect`: the `if args.no_protect` branch
  at line 1693 skips the `config.as_ref().and_then(|c| c.fallback)` lookup
  entirely — correct.

### Fail-fast action parsing

Action parsing now occurs before the scan (line 1752). Unknown actions and missing
move destinations fail before any I/O. Behavior change: previously an unknown
action would fail after the scan; now it fails before. No existing test depended on
the scan completing before the action error, and the exit code is unchanged (1).
The integration test `cli_unknown_action_errors` passes.

### Cross-group ordering

`plans.sort_by_key(|plan| plan.victim_bytes())` is applied after `build_group_plans`
(line 1825). Sorting is ascending by victim bytes, so smaller-victim groups are
listed first. No test depends on group ordering (no test asserts the order of
multiple groups). The sort is deterministic (victim_bytes is a computed u64;
stability is the Rust sort's guarantee). Benign.

### Test quality

- `test_group_plan_display_action_metrics_agree_under_oldest`: directly tests the
  TOCTOU invariant, uses explicit mtimes for determinism, verifies filesystem state.
  Meaningful.
- `cli_default_oldest_keeps_oldest_over_lexical`: pins CLI survivor identity under
  the live default, uses explicit mtimes, asserts both `KEPT` label and filesystem
  state. Meaningful.
- `cli_no_protect_lexical_reverts_to_root_index`: verifies neutral policy label and
  root_index ordering. Read-only; filesystem outcome covered elsewhere.
- Divergence-agreement test (`cli_divergent_two_metric_summary`): verifies both
  metrics are present and the protected-and-kept count. Does not assert which file
  was kept, but that is acceptable (all protected copies survive by design).
