## Verdict

The feature is **functionally complete and correct**. Every HLD §3/§4/§5 behavior I
could exercise is genuinely wired (verified by running the real binary, not just
reading code): convention default protects `00-*`/`0*`, fallback `oldest` decides
identity, `--no-protect` reverts to lexical, config files load and are honored,
`deny_unknown_fields` rejects typos with exit 1, bad globs exit 1, all-protected
groups are a verified no-op, the two-metric summary diverges correctly.

The gaps are **test-coverage gaps, not behavior gaps** — three HLD-listed behaviors
that work end-to-end but are pinned only by unit tests that bypass the real binary
(injected `base_dir` / direct `resolve_keep_policy` calls), leaving the CLI wiring
of config-file discovery and the destructive all-protected path unpinned against
regression. For a destructive tool shipping an opinionated default, those are worth
closing. Nothing is broken, faked, or `todo!()`.

## Genuine gaps / risks

1. **[SHOULD-ADDRESS] No CLI/integration test loads a real `.mddedupe.toml` through
   the binary** (HLD §4.2 resolution order; §9 test 13 "Config load + flag
   override"). Config loading is covered only by `resolve_keep_policy` unit tests
   that inject a controlled `base_dir`, deliberately bypassing the binary's actual
   `resolve_keep_policy(&args, Path::new("."))` CWD-based auto-discovery
   (`src/main.rs:1725`). So the wiring of `./.mddedupe.toml` discovery and
   `--config <valid file>` *through the real process* is untested. I verified
   manually that it works (valid `--config` with `protect-dir=["keepme*"]` correctly
   produced `KEPT (protected: dir keepme)` and `-> remove` on the sibling), but a
   regression in the `run_app` config hookup would not be caught by any test. The
   HLD's own test plan (§9 #13) calls for this via `assert_cmd`.

2. **[SHOULD-ADDRESS] `deny_unknown_fields` is not exercised end-to-end via the CLI**
   (HLD §5.8; §9 test 16 "Bad config … fatal"). The only CLI config-error test,
   `cli_bad_config_path_exits_non_zero` (`tests/cli.rs:876`), feeds a *missing* file
   → `ConfigUnreadable`. The `ConfigParse`/unknown-field rejection is covered only by
   the `resolve_keep_policy` unit test `test_resolve_keep_policy_unknown_field_rejected`
   (`src/main.rs:7511`). I verified manually that a typo'd `protect-dirs` key through
   the binary yields "Failed to parse config file … unknown field `protect-dirs`,
   expected one of …" and exits 1 — works, but unpinned at the CLI layer. The
   prompt explicitly called this out.

3. **[SHOULD-ADDRESS] The destructive all-protected group (0 victims) is not pinned
   via the CLI** (HLD §3.4b "every copy protected → untouched"; §9 test 5). Unit
   coverage exists (`test_metrics_all_protected_zero_removable`, `src/main.rs:7207`;
   `test_select_survivors_all_protected_zero_victims`, `:6847`), and
   `cli_divergent_two_metric_summary` covers a *mixed* group — but **read-only only**.
   No integration test runs `--action delete` on an all-protected tree and asserts
   nothing was removed. I verified manually (`--action delete --force` on a
   `00-a/`+`00-b/` tree): both files survive, "0 removable … 1 protected and kept",
   "Successes: 0 / 0" — correct. Closing risk: an all-protected delete is exactly the
   scenario where a wiring bug would silently destroy a user's only marked copies.

4. **[QUESTION] Trash/move actions are not tested under the protect convention.**
   The only destructive protect test through the binary is
   `cli_default_convention_protects_master_copy` (`tests/cli.rs:646`), which uses
   `delete`. `process_duplicates` is action-agnostic over the same `GroupPlan`
   victims (so the risk is low), but the HLD explain output for trash/move under
   protect (`-> trash`/`-> move` next to `KEPT (...)`) has no test. Low priority —
   flagging for completeness, not as a real hole.

5. **[QUESTION] Divergent two-metric JSON is not asserted via the CLI.**
   `cli_json_summary_outputs_valid_json` (`tests/cli.rs:111`) checks only the
   *equal* case (`removable_files == duplicate_files`). The divergent JSON
   (`removable_files < duplicate_files`, `reclaimable_bytes < duplicate_wasted_bytes`)
   is unit-tested for the text summary and field derivation, but the JSON-divergent
   path is not pinned end-to-end. Low risk — the JSON fields derive from the same
   `plans` as the (tested) text summary.

## Bottom line

Not a single behavioral hole found; the implementation faithfully delivers HLD rev. 2
including the harder corners (multi-survivor accounting, type-level
display/action/metric agreement, protect-aware collapse, neutral `--no-protect`).
The only genuine shortfalls are **three HLD-test-plan items (§9 #13, #16, #5) that are
unit-tested but not pinned through the real binary** — config-file load, CLI-level
malformed-config rejection, and a destructive all-protected no-op. All three work
when run by hand; closing them is hardening against regression in a destructive tool,
not fixing a defect. **~98% complete; the remaining ~2% is integration-test coverage,
not missing functionality.**
