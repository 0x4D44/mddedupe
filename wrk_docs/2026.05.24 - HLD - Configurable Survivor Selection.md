# HLD — Configurable Survivor Selection

- **Date:** 2026-05-24
- **Component:** `mddedupe`
- **Status:** Proposed (rev. 2 — defaults-on convention, two-metric accounting,
  protect-aware identity collapse)
- **Author:** Arthur + Claude

## 1. Summary

Today the file that survives a duplicate group is chosen by a single fixed rule
(`survivor_cmp`, `src/main.rs:36`): keep the copy under the earliest-listed scan
path (`root_index`), ties broken alphabetically. That is the right *default*
ordering but the wrong *only* option — the user can't say "always keep the copy
in my curated folders."

This change makes survivor selection **user-configurable**:

- **Protect rules** — files in priority directories (dir names starting
  `0`/`00`) or with priority filenames (`00-…`, `00 - …`) are **never** deleted.
  These are markers the user already uses to flag "this is the master."
- **A fallback strategy** — for groups where no file is protected, choose the
  keeper by `oldest` / `newest` / `shortest` path / `lexical`.

Rules live in a `.mddedupe.toml` config file (set a standing convention once)
and can be overridden per-run by CLI flags.

**The user's `0`/`00-` marker convention is the built-in default** (§3.3): with
no config file and no flags, the tool protects `0*` directories and `00-*` /
`00 - *` filenames and uses `oldest` as the fallback. A neutral policy (no
protect, `lexical` fallback — i.e. today's exact behavior) remains available via
`--no-protect` and is what `KeepPolicy::default()` yields for tests.

## 2. Goals / Non-Goals

### Goals
- Protect files by directory-name glob and by filename glob; protected files are
  never acted on.
- Support **multiple survivors per group**: if several copies are protected,
  they all stay; only unprotected copies are removed.
- Pick the **fallback keeper** for unprotected groups.
- Persist rules in a config file; allow CLI override and a clean opt-out.
- Ship the user's `0`/`00-` convention as the **default** behavior, while keeping
  a neutral (today-equivalent) policy reachable for opt-out and tests.
- Report duplicates honestly under protect (two-metric accounting, §3.5).
- Show *why* each file was kept (explain output) — this drives deletion.

### Non-Goals
- No migration, rollback, or backwards-compat shims (development sandbox).
- **No environment variables and no feature flags** — config file + CLI flags
  only.
- No scoring/weighting engine. Selection is an ordered, deterministic chain, not
  a points system (see §3.1).
- No per-rule "prefer" (rank-and-collapse) mode in this revision. Both marker
  kinds are **protect** (exclude-from-action) per the agreed semantics. The
  design leaves room for a later `prefer` mode (§8).
- No change to the move/trash/delete actions themselves, nor to scanning/hashing.

## 3. Behavior

### 3.1 Selection model ("protect, then fall back")

Each SHA-256 hash group is resolved independently. Replace the current
"sort by `survivor_cmp`, keep `[0]`, act on the rest" with **partition, then
select**:

1. **Partition** the group into `protected` / `unprotected`. A file is protected
   if it matches **any** protect rule:
   - **dir rule** — *any ancestor path component* matches a configured dir-glob
     (so a file deep under `00-best/sub/` still qualifies);
   - **name rule** — the file's own name matches a configured name-glob.
   - Matching is **case-insensitive** and is performed against a **single path
     component string** (not a path fragment) — see §5.7.
2. **If `protected` is non-empty** → survivors = *all* protected files;
   victims = *all* unprotected files. If **every** copy is protected, the group
   is left completely untouched (0 victims).
3. **If `protected` is empty** → survivors = the single best file by the
   **fallback chain**; victims = the rest.

**Fallback chain** (total order, always deterministic):

```
root_index  →  <fallback strategy key>  →  path
```

- `root_index` stays first, preserving "earliest-listed scan path wins" from the
  Multi-Drive HLD. In a single-path scan it is constant, so the strategy key
  dominates.
- strategy key: `oldest` → earliest mtime; `newest` → latest mtime;
  `shortest` → **fewest path components**; `lexical` → empty (path alone).
- `path` is the final tiebreak, so results are reproducible (and so a `shortest`
  / `oldest` tie is resolved deterministically).

### 3.2 Dir-vs-name ordering

Both marker kinds are protect, so a file matching *either* is protected
regardless of order — ordering does **not** change which files are deleted. It
only decides the **explain label** when a file matches both: the dir rule is
reported first (`protected: dir …`), per "directory priority comes first."

### 3.3 Default policy (the built-in convention)

There are two distinct "defaults", and keeping them separate is the key design
decision that lets the tool ship opinionated **without** churning the test suite:

| | Protect rules | Fallback | Where it comes from |
|---|---|---|---|
| **`KeepPolicy::default()`** (type default) | *none* | `lexical` | Rust `Default` impl. Reproduces today's `survivor_cmp` exactly (`root_index → ∅ → path`). Used by unit tests and as the `--no-protect` policy. |
| **CLI-resolved default** (no config, no flags) | `protect-dir=["0*"]`, `protect-name=["00-*","00 - *"]` | `oldest` | Built in one place during policy resolution in `run_app` (§5.4). What a bare `mddedupe <dir>` actually uses. |

Rationale:
- The user's `0`/`00-` convention is what they want out of the box, so it is the
  *CLI* default.
- Existing **unit** tests call `find_duplicates_*` / `process_duplicates`
  directly with `KeepPolicy::default()` → neutral → unchanged behavior.
- Existing **integration** tests (via the binary, so the CLI default applies)
  are safe because (a) no fixture path matches `0*` / `00-*`, and (b) the one
  survivor-identity test (`cli_read_only_lists_survivor_first`) puts its copies
  in *different roots*, so `root_index` decides the survivor before the `oldest`
  strategy is ever consulted. Any single-root survivor-identity test added later
  must set explicit mtimes (§5.6).

Opt-out: `--no-protect` selects `KeepPolicy::default()` for the run;
`protect-dir = []` / `protect-name = []` in config disables persistently.

### 3.4 Worked examples

All files below are byte-identical (same hash group).

**(a) Protect by directory** — default `protect-dir = ["0*"]`, single scan path

| File | Protected? | Outcome |
|------|-----------|---------|
| `D:\pix\00-keep\img.jpg` | yes (dir `00-keep`) | **KEPT** (protected: dir `0*`) |
| `D:\pix\2020\img.jpg`    | no | removed |
| `D:\pix\tmp\img copy.jpg`| no | removed |

**(b) Multiple protected copies all survive**

| File | Protected? | Outcome |
|------|-----------|---------|
| `D:\00-a\x` | yes | **KEPT** (protected: dir `0*`) |
| `D:\00-b\x` | yes | **KEPT** (protected: dir `0*`) |
| `D:\c\x`    | no  | removed |

This group contributes **2** to *redundancy* (3 copies − 1) but only **1** to
*removable* (one victim). See §3.5.

**(c) No protection → fallback `oldest`**

| File | mtime | Outcome |
|------|-------|---------|
| `D:\a\img.jpg` | 2019-03 | **KEPT** (oldest) |
| `D:\b\img.jpg` | 2021-08 | removed |

### 3.5 Accounting: two metrics

Today `duplicate_count` (len−1) silently equals *both* "duplicate copies that
exist" and "files that will be removed", because there is always exactly one
survivor. Protect breaks that equivalence (a group can keep >1 survivor), so the
tool reports **two** clearly-named metrics everywhere (text summary, JSON,
read-only and action paths):

| Metric | Definition | Notes |
|--------|-----------|-------|
| **redundancy** — `duplicate_files`, `duplicate_wasted_bytes` | `len − 1` per group, and the bytes thereof | "How much duplication exists." Meaning unchanged from today. |
| **removable** — `removable_files`, `reclaimable_bytes` | number of **victims** per group, and their bytes | "What this run will actually remove / reclaim." Drives `ProcessReport.total_candidates`. |

When the two diverge (protect retained extras), both are shown, e.g.:

```
Duplicate scan summary: 1000 files scanned, 12 duplicate files (840 MB);
9 removable (620 MB reclaimable), 3 protected and kept, in 4 sec.
```

In the common case (≤1 protected per group) the two coincide and the summary
reads as it does today.

## 4. CLI / Output / Config Changes

### 4.1 New flags

| Flag | Type | Meaning |
|------|------|---------|
| `--protect-dir <glob>`  | repeatable `Vec<String>` | protect files under a dir whose **name** matches `<glob>` |
| `--protect-name <glob>` | repeatable `Vec<String>` | protect files whose own **name** matches `<glob>` |
| `--no-protect` | `bool` | disable all protect rules for this run (neutral policy) |
| `--keep <oldest\|newest\|shortest\|lexical>` | clap `ValueEnum`, `Option` | fallback strategy |
| `--config <path>` | `Option<PathBuf>` | explicit config file location |

No existing flags change. No environment variables are introduced.
`--no-protect` conflicts with `--protect-dir`/`--protect-name` (clap
`conflicts_with`) so intent is never ambiguous.

### 4.2 Config file

TOML, hand-editable. The built-in default is equivalent to this file:

```toml
[keep]
protect-dir  = ["0*"]              # any ancestor dir starting with 0 / 00
protect-name = ["00-*", "00 - *"]  # filenames flagged as master copies
fallback     = "oldest"            # keeper when nothing in a group is protected
```

**Resolution order** (first present wins as the source):
1. `--config <path>` (error if given but unreadable);
2. `./.mddedupe.toml` in the current directory, if present;
3. none → the **built-in convention default** (§3.3), *not* an empty policy.

`.mddedupe.toml` is resolved relative to the current directory, independent of
the scanned paths. A home-level config is a possible future addition (would
require resolving the home directory; deferred to avoid env-var-style lookups).

**Flag/config merge** (per field; flags win):
- `--no-protect` ⇒ both protect lists are empty regardless of config.
- else if `--protect-dir` / `--protect-name` is supplied, its values **replace**
  that list; otherwise the config (or default) list is used.
- `--keep` overrides `fallback`; else config; else `oldest`.
- A config that omits a `[keep]` field falls back to the built-in default for
  that field. To get an empty list, write `protect-dir = []` explicitly.

### 4.3 Output (explain)

Both the read-only listing and the action listing print survivors **first**,
each tagged with its reason, then victims:

```
Duplicate group (hash: a1b2…):
  D:\pix\00-keep\img.jpg (1.2 MB)   -> KEPT (protected: dir 0*)
  D:\pix\2020\img.jpg (1.2 MB)      -> delete
  D:\pix\tmp\img copy.jpg (1.2 MB)  -> delete
```

In read-only mode the right column shows what *would* happen. The two code paths
share one selection function so the displayed and executed decisions can never
drift (the existing `survivor_cmp` invariant, generalized). Reason output
respects existing verbosity gating (`log_output` / `summary_only` / `quiet`,
`src/main.rs:1137`): nothing extra is printed in quiet / summary-only modes.

## 5. Design Detail (`src/main.rs`)

### 5.1 Policy types

```rust
#[derive(Clone, Copy, Debug, PartialEq, Eq, ValueEnum)]
enum FallbackStrategy { Oldest, Newest, Shortest, Lexical }

struct KeepPolicy {
    protect_dir:  GlobSet,   // matched against each ancestor dir component name
    protect_name: GlobSet,   // matched against the file name
    fallback:     FallbackStrategy,
}

impl Default for KeepPolicy {            // NEUTRAL: today's behavior
    fn default() -> Self {
        Self { protect_dir: GlobSet::empty(),
               protect_name: GlobSet::empty(),
               fallback: FallbackStrategy::Lexical }
    }
}

enum SurvivorReason { ProtectedDir, ProtectedName, Fallback }
```

`GlobSet` (the `globset` crate) compiles each rule list once, built with
`case_insensitive(true)`.

### 5.2 Selection function (single source of truth)

```rust
/// Splits a duplicate group into survivors (kept, with reason) and victims
/// (to act on). Replaces the `survivor_cmp` + `skip(1)` pattern at every call
/// site (process_duplicates, the read-only display, and the metric counters).
fn select_survivors<'a>(
    group:  &'a [DuplicateEntry],
    policy: &KeepPolicy,
) -> (Vec<(&'a DuplicateEntry, SurvivorReason)>, Vec<&'a DuplicateEntry>);
```

- Partition by the protect rules (§3.1). A file is `ProtectedDir` if any ancestor
  component matches `protect_dir`; else `ProtectedName` if its name matches
  `protect_name`. Dir tested first so it wins the label (§3.2).
- If nothing is protected, sort `unprotected` by the fallback chain and take the
  first as the sole `Fallback` survivor.
- With `KeepPolicy::default()` this yields exactly `survivor_cmp`'s survivor and
  `skip(1)` victims — proving the neutral path is unchanged.

### 5.3 Fallback comparator and mtime

`mtime` is needed **only** for `Oldest` / `Newest`, and only for groups with no
protected file. Duplicate groups are tiny, so mtime is read **on demand** at
selection time via `fs::metadata(path).and_then(|m| m.modified())` — no need to
thread a timestamp through `DuplicateEntry`, `size_map`, or every test literal.

A file whose mtime cannot be read is treated as **newest** (loses the "oldest"
contest), with the final `path` tiebreak guaranteeing a total order regardless.
Read failures are surfaced on the error log when `error_logs` is set.

### 5.4 Policy resolution and wiring

`run_app` builds the policy once, in a dedicated `resolve_keep_policy(args)`:

1. start from config (`--config` > `./.mddedupe.toml` > **built-in convention**);
2. apply flag overrides / `--no-protect` (§4.2);
3. compile globs → `KeepPolicy`.

A malformed config, an unreadable `--config` path, or an invalid glob is a fatal
startup error (new `AppError` variants → exit `1`): we refuse to run a
destructive action under a misunderstood policy. The policy is then passed by
reference to:

- `process_duplicates(duplicates, action, policy, info_logs, error_logs)` — acts
  on `select_survivors(...).victims`; `ProcessReport::new(removable_total)`.
- `find_duplicates_in_dirs(..., policy)` — computes both metric pairs (§5.5).
- the read-only display block — survivors+reasons, then victims, then metrics.

### 5.5 Accounting (two metrics)

`find_duplicates_in_dirs` currently computes `duplicate_count` / `wasted_space`
as `len − 1` (`src/main.rs:659`). It now computes **both** pairs from
`select_survivors`:

- redundancy: `Σ (len − 1)` and `Σ (len − 1)·size` (unchanged formula);
- removable: `Σ victims.len()` and `Σ victim sizes`.

`process_duplicates`' `overall_total` (`src/main.rs:1031`) becomes the removable
total, so the `Successes: X / Y` denominator and JSON `total_candidates` reflect
what is actually acted on. `JsonSummary` gains `removable_files` and
`reclaimable_bytes` alongside the existing `duplicate_files` /
`duplicate_wasted_bytes`.

### 5.6 Protect-aware identity collapse (correctness fix)

`collapse_group_by_identity` (`src/main.rs:394`) reduces a hash group so each
physical file (`FileId`) appears once, choosing the kept alias by `survivor_cmp`.
This is **not protect-aware**: a physical file reachable both through a protected
`00-*` directory and through a plain alias (symlink/junction) could be collapsed
to the *plain* representative and then deleted — defeating the protect guarantee
the markers exist to provide.

Fix: collapse must **prefer a protected alias** as the surviving representative.
The simplest correct rule, given the policy is available at this point: among
aliases sharing a `FileId`, keep a protected one if any exists; otherwise keep
the `survivor_cmp`-first. This composes protect with the existing aliasing safety
net rather than letting them fight.

### 5.7 Glob matching mechanics

Globs match a **single path-component string**, case-insensitively:

- `protect_dir` is tested against each ancestor directory's *name*
  (`Path::components` → `OsStr` → `to_string_lossy`), so separators (`\` vs `/`)
  are irrelevant and a glob never spans directory levels. Consequence:
  `--protect-dir "photos/00*"` does **not** work — pass a single component glob
  like `00*`.
- `protect_name` is tested against `path.file_name()`.
- Non-UTF-8 components are matched via `to_string_lossy` (lossy, but these are
  rare and the fallback chain still guarantees a deterministic result).

### 5.8 Config loading

```rust
#[derive(Deserialize, Default)]
#[serde(rename_all = "kebab-case", deny_unknown_fields)]
struct ConfigFile { keep: Option<KeepConfig> }

#[derive(Deserialize, Default)]
#[serde(rename_all = "kebab-case", deny_unknown_fields)]
struct KeepConfig {
    protect_dir:  Option<Vec<String>>,   // None = use built-in default list
    protect_name: Option<Vec<String>>,   // None = use built-in default list
    fallback:     Option<FallbackStrategy>,
}
```

`Option<Vec<_>>` distinguishes "field omitted" (use default) from "field set to
`[]`" (explicitly empty). Parsed with the `toml` crate; `deny_unknown_fields`
turns typos into fatal errors rather than silent no-ops.

## 8. Future-proofing (not built now)

`SurvivorReason` and the partition step leave room for a later **`prefer`** mode
(rank-and-collapse instead of exclude) without reshaping the flow: a prefer rule
would contribute an ordering key to the fallback chain rather than moving files
into the `protected` set. Out of scope here.

## 9. Testing

Driven test-first; `cargo test` / `clippy` / `fmt` stay green. New tests
(unit on `select_survivors`/`resolve_keep_policy`, integration via
`assert_cmd`/`assert_fs`):

1. **Protect by dir** — a copy under `00-keep/` survives; siblings removed.
2. **Protect by name** — `00 - master.txt` survives over plain copies.
3. **Both markers, dir-first label** — a file matching both reports
   `protected: dir …`.
4. **Multiple protected survive** — two copies under `0*` dirs both kept; the
   one unprotected copy removed (proves multi-survivor + accounting).
5. **All protected → no-op** — every copy protected; nothing deleted; removable
   count `0`, redundancy count `> 0`.
6. **No match → oldest** — unprotected group keeps the earliest-mtime file
   (**mtimes set explicitly via `filetime`**).
7. **`--keep lexical` / `shortest`** — fallback switch changes the keeper.
8. **Case-insensitive match** — `00-FOO` matches `00-*`.
9. **Ancestor match** — file deep under `…/00-best/sub/x` is protected by `0*`.
10. **Determinism** — equal mtime falls through to `path`; stable across runs.
11. **`--no-protect`** — neutral policy; survivor reverts to `(root_index, path)`.
12. **Default convention is on** — a bare run on a tree with a `00-*` file keeps
    that file without any config/flags.
13. **Config load + flag override** — `.mddedupe.toml` applies; a CLI flag
    replaces the corresponding config field; `protect-dir = []` disables.
14. **Two-metric summary** — redundancy and removable both reported and diverge
    correctly when a group retains protected extras.
15. **Protect-aware collapse (unix)** — a physical file aliased into a protected
    dir is kept via the protected representative, never deleted (§5.6).
16. **Bad config / bad glob** — fatal startup error; **nothing** on disk modified.

Existing unit tests pass `KeepPolicy::default()` (neutral) and assert identical
behavior; existing integration tests are unaffected (§3.3).

## 10. Risks

- **Opinionated default** (§3.3): a bare run now protects `0*` dirs and `00-*`
  names, so a config-less `--action delete` will keep marked files. This is
  intended, but it is a behavior change from today and is made visible by the
  explain output and the two-metric summary. `--no-protect` restores today's
  behavior. **Glob breadth:** `0*` also matches `007…`, `0day`, etc.; tighten via
  config if the user has non-marker `0`-directories.
- **Protect keeps more than one copy by design** (§3.4b). Mitigated by explain
  output making every KEPT decision and its reason visible.
- **Accounting**: the `len − 1` assumption is wired into several totals; the
  two-metric rework must touch all of them. Covered by tests 4, 5, 14.
- **Identity-collapse vs protect** (§5.6): the headline safety property; covered
  by test 15. Also note this invalidates the Multi-Drive HLD §5.2 assumption that
  there is "exactly one survivor per group."
- **mtime granularity**: handled deterministically (treat unreadable as newest,
  `path` tiebreak) and tested with explicit `filetime` mtimes; covered by 6, 10.
- **New dependencies** (`globset`, `toml`, dev `filetime`): standard crates,
  judged worth it over hand-rolled glob/TOML parsing.
