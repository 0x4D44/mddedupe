## Issue 1 — The dangerous default is under-sold [MISLEADING]

**What the docs say (readme.md, docs/usage.md):** A bulleted description of the built-in convention states `0*` protects any directory component whose name starts with `0`. The claim is technically accurate in one sentence, but neither doc makes the operational consequence visceral: a plain `mddedupe --action delete <dir>` will **silently skip deleting** duplicates under directories named `007`, `0day`, `0`, or any path whose single component begins with `0`. A user who has `0-whatever` directories by accident — naming convention unrelated to mddedupe — gets surprise survivors with no warning. The docs describe the feature and its default, but they never say "if you do not intend to use the convention, pass `--no-protect`." There is no safety callout box, no "WARNING" admonition, no example of the silent-keep case.

**Code confirmation:** `DEFAULT_PROTECT_DIR = &["0*"]`, glob is case-insensitive, applied to every single ancestor component (line 98, `any_ancestor_dir_matches`). The `0*` glob matches any single string starting with `0`: `0`, `007`, `0day`, `0x4D44`, all protected silently.

**Risk:** A user running `--action delete` on a real photo archive that has folders named `0001-Jan`, `0002-Feb`, etc. (a common convention) will find those duplicates are never deleted. Nothing in the output except the `KEPT (protected: dir 0001-Jan)` line per file signals this — and if `--quiet` or `--summary-only` is set, even that is hidden.

**Required fix:** Add an explicit "important" note in the "Built-in convention" subsection stating that `0*` is breadth-first (matches any directory component beginning with digit zero, including numeric-only names and sequences like `007`), and instruct users who do not follow this naming convention to pass `--no-protect`.

## Issue 2 — `readme.md` Safety Features still says "first file … always kept" [MISLEADING]

**Location:** `readme.md`, Safety Features section, item 4:

> **Original preservation**: First file in each group always kept

This line was **not updated** in the diff. It is factually wrong under the new feature. With the built-in default policy, the file kept is the one protected by a glob rule, not the first file alphabetically. With `--keep oldest`, the oldest file is kept regardless of alphabetical order. The claim "always kept" is only true for the neutral lexical policy (`--no-protect`), not the default.

A user reading the Safety Features bullet list to decide whether `--action delete` is safe will be misled into believing alphabetical-first always survives, which is false.

Also at line 294 of `readme.md`:

> 7. **Action Execution**: Apply move/trash/delete to all but first file in each group

This is a leftover from the pre-feature architecture description. It has the same problem: "all but first" only describes the `lexical/root_index` fallback; protect rules change which copies are victims entirely.

## Issue 3 — `--no-protect` conflict claim is over-stated [SHOULD-ADDRESS]

**What docs/usage.md says:**

> Conflicts with `--protect-dir` / `--protect-name`.

**What the code actually enforces (line 433):**

```rust
#[arg(long, conflicts_with_all = ["protect_dir", "protect_name"])]
no_protect: bool,
```

That is correct. But the same table cell also says:

> An explicit `--keep` still overrides the neutral fallback when combined with `--no-protect`.

This implies `--keep` is compatible with `--no-protect`, which the code confirms (no `conflicts_with` between them). However, the sentence is in the `--no-protect` row, not its own row, so a reader scanning the table for `--keep` semantics will not see it. Meanwhile the `--keep` row has no mention of its interaction with `--no-protect` at all. The interaction — "clears protect lists but you can still pick oldest/newest/etc." — is the key escape hatch for the dangerous-default problem raised in Issue 1, and it is buried in a subordinate clause.

## Issue 4 — Sample output lines use wrong verb for read-only mode [MISLEADING]

**docs/usage.md**, `KEPT (reason)` section, sample output line 4:

```
/data/pix/tmp/photo.jpg  (1.2 MB) -> remove
```

**What the code actually prints** in read-only mode (`would_verb`, line 1120–1124): `remove`. For an action run (e.g. `--action delete`), it would print `delete`. The sample shows `remove`, which is only correct for the read-only listing. The sample is immediately under a section called `KEPT (reason) explain output` and shows no action flag, so `remove` is technically right for that context.

However, the `readme.md` example (lines 308–311) does the same:

```
D:\pix\2024\photo.jpg      -> remove
D:\pix\tmp\photo copy.jpg  -> remove
```

Neither document clarifies that `remove` is the **preview verb** for read-only mode and that a real `--action delete` run prints `delete`, `trash`, or `move` instead. A user who sees `remove` and then does an actual delete run is not misled about outcomes, but the docs create a subtle expectation mismatch that a conscientious user will notice and distrust.

This is a minor issue but worth a one-line clarification ("The verb changes to `delete`, `trash`, or `move` when an action is requested").

## Issue 5 — ARCHITECTURE.md `KeepPolicy::default()` description is subtly wrong for `--no-protect` [SHOULD-ADDRESS]

**ARCHITECTURE.md** (new section, paragraph 2):

> `KeepPolicy::default()` is the **neutral policy**: empty GlobSets and `Lexical` fallback. This reproduces the pre-feature `survivor_cmp` behavior exactly (`root_index → path`) and is what `--no-protect` activates.

**Code:** `--no-protect` does NOT directly create a `KeepPolicy::default()`. It calls `resolve_keep_policy` which, when `args.no_protect == true`, sets both protect lists to `Vec::new()` and sets `fallback = args.keep.unwrap_or(FallbackStrategy::Lexical)`, then calls `compile_protect(&[])` for each and returns a new `KeepPolicy`. The resulting struct is equivalent to `KeepPolicy::default()` only when no `--keep` flag is also passed. When `--no-protect --keep oldest` is used, the resulting policy has `fallback = Oldest`, which is NOT `KeepPolicy::default()`.

The statement "is what `--no-protect` activates" conflates the conditional case (`--no-protect` alone) with the unconditional type default. A contributor reading this may incorrectly assume that `--no-protect --keep oldest` still produces `KeepPolicy::default()` and be confused when the test fails or the behavior differs.

The CLAUDE.md duplicate of this claim has the same problem in slightly different wording:

> `KeepPolicy::default()` is the neutral policy (empty GlobSets, `Lexical` fallback) — used by unit tests, not by the CLI.

That sentence is accurate by itself, but the paragraph below says "`--no-protect` activates" the neutral policy, again without the caveat that `--keep` changes the fallback dimension even under `--no-protect`.

## Issue 6 — Glob footgun: no mention that `0*` matches literal `0` (zero-length suffix) [QUESTION]

The docs explain single-component matching and the `photos/00*` footgun. They do not mention that `0*` (glob wildcard, zero or more chars) also matches a directory named exactly `0` — a single-character directory name. This is likely intentional and correct, but users who write custom patterns like `--protect-dir "archive*"` will correctly protect `archive` itself (not just `archive-2024`), and there is no callout that `*` can match zero characters. A user who writes `--protect-dir "0?"` intending "starts with 0 and one more char" would correctly use `?` for exactly-one, but there is no guidance on `?` vs `*` semantics in the docs. Minor, but a one-liner note that `*` matches zero or more characters (so `0*` includes `0` itself) would prevent confusion.

## Verdict

The docs are largely accurate and internally consistent. The two issues that would actually burn a user are: **(1)** no prominent warning that the default `0*` protect rule will silently keep files the user expects to be deleted — especially on numeric-prefix directory conventions — and **(2)** the stale "first file always kept" claim still sitting in the readme Safety Features section, which directly contradicts the new feature's semantics and will mislead anyone using it to reason about safety.
