## UNCLEAR (should-fix)

### U1 ‚Äî ARCHITECTURE.md: "`--no-protect` activates `KeepPolicy::default()`" is an oversimplification

**File:** `ARCHITECTURE.md`, Survivor Selection ‚Üí `KeepPolicy` and `FallbackStrategy`

> "This reproduces the pre-feature `survivor_cmp` behavior exactly (`root_index ‚Üí path`) and is what `--no-protect` activates."

The code is:
```rust
let fallback = if args.no_protect {
    args.keep.unwrap_or(FallbackStrategy::Lexical)
};
```

`--no-protect` alone yields `KeepPolicy::default()` (empty GlobSets + `Lexical`). But `--no-protect --keep oldest` yields empty GlobSets + `Oldest`, which is NOT `KeepPolicy::default()`. The sentence implies `--no-protect` always activates the neutral default policy, which is false when `--keep` is also passed. The sentence should say "is what `--no-protect` (without `--keep`) activates" or split the two cases.

The same text appears (correctly qualified) in `docs/usage.md` and `CLAUDE.md` with "(unless `--keep` overrides)" ‚Äî only `ARCHITECTURE.md` omits the qualification.

### U2 ‚Äî `readme.md`: "today's pre-feature behavior" is stale phrasing

**File:** `readme.md`, Survivor Selection flags block

> `--no-protect` - Disable all protect rules for this run; uses `lexical` fallback (today's pre-feature behavior), unless `--keep` overrides.

The `--help` text says "today's behavior" (no "pre-feature"). Calling it "pre-feature" in the user-facing readme implies the feature is not yet shipped. Since the docs are part of the feature release, this should be "pre-feature behavior" ‚Üí "original behavior" or drop the parenthetical entirely. (The `--help` itself says `(today's behavior)` without "pre-feature", and that is the canonical wording.)

## NIT

### N1 ‚Äî `docs/usage.md`: precedence table row ordering is misleading for fallback

**File:** `docs/usage.md`, Precedence / resolution order table

The table lists `--no-protect` (row 2) before `--keep` (row 3) in the "first wins" order. For the fallback column, `--keep` actually overrides `--no-protect`'s default of `Lexical`. The cell text says "forces `lexical` (unless `--keep` overrides)" which is correct, but a reader skimming row order might conclude `--no-protect` beats `--keep` for fallback. Consider swapping the rows for `--no-protect` and `--keep`, or add a footnote explaining that `--keep` and `--no-protect` interact (rather than being strictly ordered).

### N2 ‚Äî `docs/usage.md`: short summary example says "in 0 sec." ‚Äî technically correct but may mislead

**File:** `docs/usage.md`, Summary metrics section

```
2 files scanned, 1 duplicate files (13 bytes wasted) in 0 sec.
```

`format_duration(Duration::ZERO)` produces `"0 sec"` so the line as shown is valid. However "0 sec" appears only in test/dev runs with instant execution; real users will see e.g. `"2 sec"`. This is fine as a doc example but worth noting ‚Äî no change needed unless Arthur wants a more realistic value like `"< 1 sec"`.

### N3 ‚Äî `ARCHITECTURE.md`: `resolve_keep_policy` fallback description ordering is awkward

**File:** `ARCHITECTURE.md`, Survivor Selection ‚Üí `resolve_keep_policy`

> **fallback** ‚Äî `--keep` if passed; else (when `--no-protect` is not set) the config `fallback`; else `--no-protect` ‚Üí `Lexical`; else `DEFAULT_FALLBACK` (`Oldest`).

The parenthetical `"(when --no-protect is not set)"` breaks the linear "first wins" reading. The actual code is a two-branch conditional on `args.no_protect`, not a sequential priority list. The description is not wrong, but it is harder to parse than a simple two-case description would be:

> **fallback** ‚Äî if `--no-protect`: `--keep` if passed, else `Lexical`. Otherwise: `--keep` if passed, else config `fallback` if present, else `DEFAULT_FALLBACK` (`Oldest`).
