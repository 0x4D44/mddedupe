## Summary

Stage 3 flips the feature ON. It delivers:

1. **Config structs** — `ConfigFile` / `KeepConfig` (TOML, `deny_unknown_fields`); `FallbackStrategy` gains `serde::Deserialize`.
2. **`load_config`** — explicit `--config` path (fatal if unreadable); auto-discovery of `./.mddedupe.toml`; `None` otherwise.
3. **`compile_protect`** — builds a `GlobSet` from a pattern list, mapping the first bad glob to a named `AppError::InvalidGlob`.
4. **`resolve_keep_policy`** — per-field merge of flags over config over built-in convention defaults (`0*` / `00-*` / `oldest`).
5. **CLI flags** — `--protect-dir`, `--protect-name`, `--no-protect`, `--keep`, `--config` added to `Args`.
6. **`run_app` flip** — replaces the hardcoded `KeepPolicy::default()` with `resolve_keep_policy(&args)`, making the convention live by default.
7. **Explain output** — `KEPT (reason)` labels for survivors in both read-only and action display paths; `process_duplicates` now partitions once and reuses the split.
8. **`would_verb` in read-only listing** — previews what would happen based on `--action` value.
9. **`format_app_error`** — three new error variants with exit code 1.
10. **New unit tests** — eight `test_resolve_keep_policy_*` tests covering every merge branch.
11. **New integration tests** — six `cli_*` tests covering default convention, `--no-protect`, flag override, two-metric summary, bad config, bad glob.
12. **Existing-test migration** — all ~40 `run_app` literal struct construction sites gain the five new flag fields (all neutral: empty vecs, `false`, `None`).

## Findings

### MUST-FIX

#### M1. `test_resolve_keep_policy_no_config_is_convention_default` is CWD-sensitive and will become flaky

**Location:** `src/main.rs` — `test_resolve_keep_policy_no_config_is_convention_default` (approx. line 7173)

**Description:** The test calls `resolve_keep_policy(&policy_args())` with `args.config = None`. Inside `load_config`, the auto-discovery path constructs `PathBuf::from(".mddedupe.toml")` and calls `.exists()` against the process CWD — which for `cargo test` is the crate root (`C:\language\mddedupe`). There is no `.mddedupe.toml` at the crate root **today**, so the test passes and hits the "no config → built-in convention" branch correctly. But:

- Anyone who creates `C:\language\mddedupe\.mddedupe.toml` for manual testing will silently break this test.
- Any future test helper that writes a `.mddedupe.toml` to a relative path without controlling CWD could land there.
- The HLD §9 testing strategy says the "default convention is on" test (item 12) should be hermetic but does not document the CWD dependency for the unit test.

`test_resolve_keep_policy_config_applies`, `test_resolve_keep_policy_flag_overrides_config`, etc., are safe because they pass an explicit `--config` path.

**Risk:** Silent correctness inversion. If `.mddedupe.toml` is ever present at the crate root, the test asserts the convention default but the policy actually comes from the file. The assertions (`protect_dir.is_match("0archive")`) would still pass if the file happened to contain `0*`, masking the real problem entirely.

**Fix:** The test should either (a) construct a temp dir, set it as the `directories` path, but — more directly — call `load_config` via `resolve_keep_policy` with a fake CWD by temporarily `set_current_dir` (fragile in a parallel test suite), or (b) use a `policy_args_with_config(None, explicit_temp_dir_path)` that passes a `--config` pointing to a guaranteed-nonexistent file so the branch that returns `None` is exercised without touching CWD, or (c) the simplest correct fix: pass `config: Some(PathBuf::from("definitely-not-a-real-config-xyz.toml"))` but that hits the "explicit config missing → fatal" path, which is wrong. The correct minimal fix is to add a `.current_dir(temp)` override in `load_config` via an injected base-path parameter, OR to separate the "no config → convention" unit test from `resolve_keep_policy` by testing `load_config` directly and confirming `None` return when no file exists at a controlled path.

The simplest non-invasive hardening that requires no refactor: rename this test to include a comment marking it CWD-sensitive, and add a `assert!(!PathBuf::from(".mddedupe.toml").exists(), "test assumes no .mddedupe.toml at crate root")` guard at the top of the test body. This makes the assumption explicit and causes an informative failure rather than a silent correctness inversion. The real fix is a structural one (inject a base path into `load_config`) but that is a larger refactor.

### SHOULD-FIX

#### S1. Double `select_survivors` in the read-only path: sort-key and display calls are independent

**Location:** `src/main.rs` — `run_app`, the `if log_output` block (approx. lines 1760–1800)

**Description:** The read-only display path calls `select_survivors` twice on each group:

1. `groups.sort_by_key(|(_, group)| { let (_survivors, victims) = select_survivors(group, &keep_policy); ... })` — used only for sort-key computation.
2. `let (survivors, victims) = select_survivors(group, &keep_policy);` — inside the per-group print loop, used for actual display.

For `Oldest`/`Newest` strategies, each call reads mtime via `mtime_of` anew. Between the two calls, a file's mtime could change (e.g., another process writes to it while a large scan is in progress). In that scenario, the survivor elected in call 1 (sort) is a different file than the survivor elected in call 2 (display), and the displayed `KEPT` label would be on a different entry than what the action path would ultimately act on. More precisely, the action path (`process_duplicates`) calls `select_survivors` a **third** time on the same data — but `process_duplicates` already unifies its count and action in a single call via `partitioned`, so that path is safe. The split is only between the sort-key call and the display call.

**Practical severity:** Low-to-medium. A file's mtime changing mid-scan is uncommon on a desktop workstation, but the window exists and the inconsistency would be confusing: the listing would show file A as `KEPT` but the action (if the user proceeds) would actually keep file B. The display exists precisely to let the user reason about what will happen — an inconsistency there is a UX defect.

**Fix:** Pre-compute `select_survivors` once per group before sorting, store the `(survivors, victims)` alongside the sort key, and reuse in the print loop. This also removes a redundant mtime read pass. Example:

```rust
let mut groups: Vec<_> = duplicates.iter()
    .filter(|(_, g)| g.len() > 1)
    .map(|(hash, group)| {
        let (survivors, victims) = select_survivors(group, &keep_policy);
        let victim_bytes: u64 = victims.iter().map(|(_, sz, _)| *sz).sum();
        (hash, survivors, victims, victim_bytes)
    })
    .collect();
groups.sort_by_key(|(_, _, _, bytes)| *bytes);
```

#### S2. `would_verb` in the read-only listing duplicates the `action_label` logic with different coverage

**Location:** `src/main.rs` — `run_app`, approx. lines 1767–1772, and `action_label` at approx. line 1082

**Description:** Two separate places map an action to a display verb:

- `action_label(action: &DuplicateAction) -> &'static str` — used in `process_duplicates` for the action path, covers `Move`/`Trash`/`Delete`.
- `would_verb` in the read-only block — a manual `match args.action.as_deref()` with three `eq_ignore_ascii_case` arms plus a fallback `"remove"`.

These can diverge if a new action type is added. Additionally, `would_verb` uses `eq_ignore_ascii_case` for string comparison while the actual action dispatch in `run_app` (at approx. line 1831) lowercases the action string first and then matches against lowercase literals. The two paths are consistent today but fragile — if the user passes `--action DELETE` (all caps), `would_verb` correctly maps it to `"delete"` via `eq_ignore_ascii_case`, but the display says "delete" while an `UnknownAction` error is never reached (since the dispatch path also lowercases). So this is not a live bug, but the duplication is a maintenance hazard.

**Fix:** Factor `would_verb` to go via `action_label` if possible, or at minimum add a comment linking the two. An alternative: parse the action string into `DuplicateAction` once upfront (before the scan) and pass it through, so the read-only block can call `action_label` directly rather than re-deriving the verb from the raw string. This would also simplify the existing switch between read-only and action modes.

#### S3. `compile_protect` performs a redundant re-scan to find the offending pattern

**Location:** `src/main.rs` — `compile_protect`, approx. lines 1593–1601

**Description:**

```rust
fn compile_protect(patterns: &[String]) -> Result<GlobSet, AppError> {
    build_globset(patterns).map_err(|err| {
        let pattern = patterns
            .iter()
            .find(|p| GlobBuilder::new(p).case_insensitive(true).build().is_err())
            .cloned()
            .unwrap_or_default();
        AppError::InvalidGlob(pattern, err.to_string())
    })
}
```

When `build_globset` fails (which only happens when `GlobBuilder::new(p).build()` returns `Err`, since `builder.build()` on a `GlobSetBuilder` cannot fail after valid individual globs are added), the error closure iterates patterns again to find the first bad one. This is both redundant (the error already came from the first bad pattern) and requires constructing a second `GlobBuilder` for each pattern until the bad one is found. In practice the list is short, so this is a performance non-issue. The real concern is correctness: if `GlobSetBuilder::build()` could ever fail for a reason other than a bad individual pattern (e.g., a future `globset` version), the retroactive search would return `unwrap_or_default()` (an empty string), producing a confusing error message `"Invalid protect glob '': ..."`.

The correct approach is to stop at the first bad pattern in `build_globset` and surface it in the error directly. This can be done by refactoring `build_globset` to return the bad pattern alongside the error, or by accumulating patterns one at a time and aborting on the first failure:

```rust
fn compile_protect(patterns: &[String]) -> Result<GlobSet, AppError> {
    let mut builder = GlobSetBuilder::new();
    for pattern in patterns {
        let glob = GlobBuilder::new(pattern)
            .case_insensitive(true)
            .build()
            .map_err(|err| AppError::InvalidGlob(pattern.clone(), err.to_string()))?;
        builder.add(glob);
    }
    builder.build().map_err(|err| AppError::InvalidGlob(String::new(), err.to_string()))
}
```

This would also eliminate the separate `build_globset` function (which is now only called by `compile_protect` and by unit tests that could call `compile_protect` directly). The current `build_globset` is still tested directly by some unit tests, so removal would require adjusting those callers, but the pattern is cleaner.

### NITs

#### N1. `survivor_reason_label` scans ancestors in reverse-path order, naming the nearest protected ancestor

**Location:** `src/main.rs` — `survivor_reason_label`, approx. lines 196–203

The function traverses path components in reverse (`path.components().rev().skip(1)`) and finds the first one (nearest to the file) that matches `protect_dir`. This means for a file at `D:\00-best\sub\img.jpg` where `protect_dir = ["0*"]` and both `00-best` and a hypothetical root `0storage` match, the label shows `"00-best"` (the nearest match), not the outermost. This is likely the desired UX behavior ("why was THIS file protected?") and the HLD does not specify a preference, but neither the code nor the HLD comment at §3.2 (which discusses only the label for dir-vs-name, not near-vs-far ancestor) documents the "nearest ancestor" choice. A one-line comment clarifying the intent would prevent future confusion.

#### N2. `policy_args()` baseline has `directories: vec![PathBuf::from(".")]`

**Location:** `src/main.rs` — `policy_args()` test helper, approx. line 1278

`policy_args()` uses `.` as the scan directory, which is fine since `resolve_keep_policy` does not actually scan. However, tests that accidentally call `run_app(policy_args(), ...)` without changing `directories` to a controlled temp path would scan the crate root, potentially finding real files. The helper is only used by `test_resolve_keep_policy_*` which all call `resolve_keep_policy` directly, so this is a NIT about future misuse. A comment on `policy_args` noting it is only for policy-resolution tests would help.

#### N3. `format_scan_summary` wording mismatch: divergent-case summary says "duplicate files" but normal case says "duplicates found"

**Location:** `src/main.rs` — `format_scan_summary`, approx. lines 1545–1565

The two branches produce slightly inconsistent phrasing:
- Normal: `"N files scanned, M duplicates found (X wasted) in T."`
- Divergent: `"N files scanned, M duplicate files (X); R removable (Y reclaimable), P protected and kept, in T."`

"Duplicates found" vs "duplicate files" is a minor inconsistency. The integration test (`cli_divergent_two_metric_summary`) asserts `"2 duplicate files"` which matches the divergent branch; the format is probably fine as-is, but aligning wording between branches would avoid future confusion when reading logs that mix both formats.

## Cross-Cutting Observations

### `resolve_keep_policy` merge correctness — assessed correct

Every case from HLD §4.2 is implemented correctly:

| Scenario | Behavior |
|---|---|
| No config, no flags | Built-in convention defaults via `default_dir` / `default_name` / `DEFAULT_FALLBACK` |
| Config present, field omitted | `config.as_ref().and_then(|c| c.protect_dir.clone())` returns `None` → falls back to `default_dir` |
| Config present, explicit `[]` | `Some(vec![])` → empty list used; default is NOT applied (correct) |
| `--protect-dir` flag present | Non-empty `args.protect_dir` replaces the config/default list |
| `--no-protect` | Both lists forced to `Vec::new()` regardless of config; fallback resolves independently |
| `--no-protect` fallback | `args.keep` or `config.fallback` or `DEFAULT_FALLBACK` — `--no-protect` does not touch this path |
| `--config` explicit path | Used unconditionally; fatal if missing/malformed |
| `./.mddedupe.toml` auto-discovery | CWD-relative lookup; correct per HLD §4.2; documented limitation re: home-level config |
| `--config` precedence over `./.mddedupe.toml` | Correct: `Some(path)` arm in `load_config` skips the auto-discovery path entirely |

The merge logic is correct for all documented cases. The CWD sensitivity is a test hygiene issue (M1), not a logic error.

### Default-flip safety audit — no missed victim/survivor inversions found

All existing unit tests that call `run_app` with `no_protect: false, keep: None, config: None` were audited:

- Tests asserting "one file remains" or "content equals X" do not assert *which* specific path survived — they only check that exactly one remains and its content is correct. Since all duplicate pairs in these tests have identical content, the `oldest` fallback picking a different file than `lexical` does not break any assertion.
- No test fixture path component starts with `0` (temp dir names from `TempDir::new()` are random hex strings on all tested platforms, not starting with `0`). The convention `0*` glob would never match.
- No test fixture file name starts with `00-` or `00 ` (all test fixtures use names like `file1.txt`, `file2.txt`, `original.txt`, `a.txt`, `b.txt`). The convention `00-*` / `00 - *` globs would never match.
- The `test_run_app_move_action_generates_unique_destination` test pre-creates `file2.txt` in the destination before the move. It checks that exactly two files exist in the destination after the move (the pre-created `file2.txt` + one renamed moved copy), and that the moved copy is NOT named `file2.txt` (collision avoidance). This is independent of which source file (`file1.txt` vs `file2.txt`) was the survivor — the test passes regardless of whether `file1.txt` or `file2.txt` is the victim. **Safe.**
- The `cli_read_only_lists_survivor_first` integration test does NOT add `--no-protect`. Under the default convention, both copies are in different roots (`dir_zzz` = root 0, `dir_aaa` = root 1), so `root_index` decides the survivor before the `oldest` strategy is ever consulted (HLD §3.3 note and §3.1 fallback chain). No `0*`-matching directory names in the fixture. **Safe.** The companion new test `cli_no_protect_lexical_reverts_to_root_index` correctly pins the neutral policy.

**One observation about `cli_read_only_lists_survivor_first`:** While this test is currently safe, it relies on the documented (but not asserted) assumption that `root_index` decides before `oldest`. If the `oldest` fallback were somehow introduced before `root_index` in a future refactor, this test would fail silently (or pass for the wrong reason). Adding a comment noting this dependency would make the safety argument auditable.

### Determinism across passes — assessed acceptable with documented assumption

`find_duplicates_in_dirs`, `run_app` read-only sort, `run_app` read-only display, and `process_duplicates` each call `select_survivors` independently. For `Oldest`/`Newest` strategies, each call reads mtime on demand. The HLD §5.3 documents: "mtime is assumed stable for the duration of the scan." This assumption is acceptable for a desktop tool (mtimes changing during a dedup scan is uncommon and the user would need to re-run anyway), but it is not documented anywhere the user sees. The double-call in the read-only display (S1) is the only place this creates a visible inconsistency (sort vs. display could produce different survivors), and is the more important issue to fix. The `find_duplicates_in_dirs` vs. `process_duplicates` calls use the same data snapshot (the `duplicates` HashMap is built once before both), so they are consistent as long as mtimes don't change between the two phases — an acceptable assumption.

### Explain output correctness — assessed correct

- **Dir-first label:** `protect_reason` tests dir before name (line 165–172) — correct.
- **Survivors-before-victims ordering:** Both the action path (survivors loop, then victims loop) and the read-only display path (same structure) print survivors first — correct.
- **Verbosity gating:** Explain output is inside `if info_logs` (action path) and `if log_output` (read-only path). Both flags exclude `quiet` and `summary_only` modes — correct. No explain output leaks to the summary text.
- **Read-only wording:** `would_verb` ("move"/"trash"/"delete"/"remove") used in read-only, `action_label` used in action path — both show what would/will happen. The `would_verb` duplication is covered by S2.

### `AppError` messages and exit codes — assessed correct

All three new variants use exit code 1. Error messages are user-readable, include the path for config errors, and include the offending glob pattern for `InvalidGlob`. The `--config` path is shown via `.display()` (correct for cross-platform output). No exit code inconsistency with existing variants.

### `deny_unknown_fields` scope — assessed correct

Both `ConfigFile` and `KeepConfig` carry `deny_unknown_fields`. A typo at either the top-level (`[kep]`) or within the `[keep]` table (`protect-dirs`) will be rejected with `AppError::ConfigParse`. The test `test_resolve_keep_policy_unknown_field_rejected` covers the inner table case. The top-level typo case (`[kep]` with no `[keep]`) deserves a test but is a NIT since `deny_unknown_fields` on `ConfigFile` guarantees it is handled.
