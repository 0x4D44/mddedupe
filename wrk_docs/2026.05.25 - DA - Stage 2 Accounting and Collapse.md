## 1. The ¬ß5.6 safety fix is effectively unverified on Windows ‚ÄĒ and not unit-isolated [BLOCKING]

¬ß5.6 is the *headline correctness property* of Stage 2 (Risk list, HLD: "the
headline safety property"). The only test that exercises it is
`test_collapse_prefers_protected_alias` at `src/main.rs:6420`, gated
`#[cfg(unix)]` because it relies on `fs::hard_link` to give two paths one
`FileId`.

**On Windows this branch is never executed by any test.** The two Windows-runnable
collapse tests (`test_collapse_group_by_identity_keeps_one_per_physical_file`
`src/main.rs:3503`, `..._preserves_distinct_and_unreadable` `src/main.rs:3525`)
both pass `KeepPolicy::default()`. With empty globs, `protect_reason` is always
`None`, so `(incoming_protected, current_protected)` is always `(false, false)`
and control always takes the `_ => survivor_cmp(...)` arm. The two new arms ‚ÄĒ
`(true, false) => true` and `(false, true) => false`, i.e. the entire fix ‚ÄĒ are
dead on Windows CI. A regression that, say, inverted those arms (preferring the
*plain* alias) would compile clean, pass every Windows test, and silently
reintroduce the exact data-loss bug the markers exist to prevent.

**The scenario the fix guards (and that needs a guard test):** one physical file,
two names ‚ÄĒ `‚Ä¶/00-keep/file` (protected by `0*`) and `‚Ä¶/plain/file` (a hardlink /
junction / symlink alias), with the *plain* alias at the lower `root_index`.
Under the neutral collapse rule `survivor_cmp` keeps the lower-`root_index` plain
entry, which then becomes a victim and gets deleted ‚ÄĒ destroying the only inode,
including the data the `00-keep` marker was protecting. The unix test constructs
exactly this and asserts the protected alias survives. Nothing equivalent runs on
the team's actual dev platform (Windows).

**Is portable coverage feasible? Yes ‚ÄĒ two routes, and at least the second is cheap:**

1. **Windows alias fixture.** On Windows `FileId` is `canonicalize()`-based
   (`src/main.rs:394`), so two paths that canonicalize to one target *do*
   collapse. A directory junction (`mklink /J`, no admin needed) pointing into a
   dir that holds the file yields two paths to one physical file that canonicalize
   identically ‚ÄĒ a genuine `#[cfg(windows)]` analogue of the unix hardlink test.
   More setup than the unix case, but real and unprivileged.

2. **Factor the decision into a pure function (preferred).** The replace decision
   (`src/main.rs:632-636`) is inline in the collapse loop and reads only two
   booleans + `survivor_cmp`. Extract:

   ```text
   fn prefer_replacement(incoming_protected, current_protected, cmp_is_less) -> bool
   ```

   and unit-test all four `(bool, bool)` combinations plus both `survivor_cmp`
   outcomes ‚ÄĒ no filesystem, no platform gate, runs everywhere. This makes the
   *protect-preference logic* (the part that regressed in my hypothetical)
   testable independently of the FS-aliasing plumbing (which the unix test can
   keep covering as an integration check). This is the higher-value fix because
   the aliasing mechanics are stable; the protect-preference policy is the new,
   fragile code.

**Recommendation:** add the pure-function unit test (route 2) as a minimum bar
before this stage is called done; route 1 is a nice-to-have integration guard.
Shipping the safety fix with zero executed assertions on the dev platform is not
acceptable for "the headline safety property."

## 2. Metric edge cases ‚ÄĒ arithmetic is correct, but rests on an unasserted invariant [QUESTION]

Walking the cases requested:

- **Entirely protected group (0 victims):** correct. `select_survivors` returns
  all-protected survivors + empty victims; `removable_files += 0`,
  `reclaimable_bytes += 0`, while `redundant_files += len-1`. Covered by
  `test_metrics_all_protected_zero_removable` (`src/main.rs:6360`-ish).

- **Collapse removes aliases before counting:** correct *ordering*. Collapse runs
  at `src/main.rs:901-903`, the `len>1` retain at `905`, and metric counting at
  `915-921` ‚ÄĒ all on the already-collapsed groups. Both metrics see post-collapse
  groups, so an aliased file is counted once in both. Good.

- **Multiple protected survivors:** correct. `removable = unprotected.len()`,
  independent of how many survivors there are.

- **"protected and kept = redundancy ‚ąí removable" ‚ÄĒ can it go negative/wrong?**
  Only if `removable > redundancy` for some group. Per group:
  `redundancy = len-1`, `removable = victims = len - survivors`, and
  `survivors >= 1` whenever the group is non-empty (a fallback always elects one
  keeper; protect elects ‚Č•1). So `removable <= len-1 = redundancy` always, and the
  subtraction (`src/main.rs:1546`, `duplicate_count - removable_files`) is
  non-negative. **However** this is a global-sum subtraction, and its correctness
  depends on `survivors >= 1` per group, which is *implied* but not asserted
  anywhere. It holds today because the summary branch is unreachable from the
  neutral CLI ‚ÄĒ but see ¬ß5: if Stage 3 flips the default and that invariant ever
  breaks (e.g. a future `prefer` mode that could elect zero survivors), this
  `usize` subtraction underflows and panics. Worth a one-line `debug_assert` or a
  saturating sub with a comment, given it's a `usize`.

- **Mixed sizes within a group:** verified the code does **not** wrongly assume
  equal sizes ‚ÄĒ it sums actual per-victim `size` for `reclaimable_bytes`
  (`src/main.rs:920`) rather than `victims.len() * group_size`. Defensively
  correct even though files are grouped by `metadata.len()` first
  (`src/main.rs:758`), so within a hash group sizes are in fact equal. No bug, but
  note `redundant_bytes` uses `skip(1)` on the *unsorted* group (`917`) while
  `reclaimable_bytes` uses victims ‚ÄĒ these only coincide because sizes are uniform.
  Fine today; fragile if grouping ever changed.

**Verdict on metrics:** no correctness bug found. The risk is documentary ‚ÄĒ
unasserted invariants (`survivors >= 1`, uniform group size) that a later stage
could violate into a panic.

## 3. Double `select_survivors` in `process_duplicates` ‚ÄĒ benign now, real Stage-3 hazard [SHOULD-ADDRESS]

`process_duplicates` calls `select_survivors` **twice per run**: once to compute
`overall_total` (`src/main.rs:1297-1301`) and again inside the action loop
(`src/main.rs:1311`). For `Oldest`/`Newest` the selection reads mtimes from the
filesystem (`select_survivors` ‚Üí `mtime_of`, `src/main.rs:196`). The two calls
happen at different instants.

At the Stage 2 neutral default this is **immune**: `Lexical` reads no mtimes, and
`overall_total` only counts `.1.len()` (victim *count*), which is order-invariant
even if mtimes shifted. So `total_candidates` (the `Successes: X / Y` denominator
and JSON `total_candidates`) cannot disagree with the actual action set today.

But the latent bug is real and lands exactly when Stage 3 turns on mtime
fallbacks: if a file's mtime changes between the two passes (or a victim is
deleted/relocated mid-loop and the count pass re-reads ‚ÄĒ it doesn't here, but the
shape invites it), the *count* pass and the *action* pass could select different
victim sets. Even the count is only safe because no protect/fallback policy can
change the *cardinality* across two reads ‚ÄĒ but `Newest`/`Oldest` with an
unreadable-vs-readable mtime flip *can* change which file is the single survivor,
and while that keeps cardinality = len-1 for a fallback group, a mixed
protect+fallback world is harder to reason about. The clean fix is to compute the
victim sets **once** and derive both the total and the actions from that single
partition, rather than calling `select_survivors` twice. This also halves the FS
reads.

**Flag for Stage 3:** not a Stage 2 blocker (neutral immunity verified), but the
"single source of truth so display and action can never drift" claim is quietly
violated by calling the source of truth twice. Compute once, use twice.

## 4. Neutral-invariance claim ‚ÄĒ holds; one nuance on the listing sort [QUESTION]

I tried to construct a neutral input where Stage 2 diverges from Stage 1:

- **CLI is hard-wired neutral** at `src/main.rs:1416` (`KeepPolicy::default()`),
  so `removable_files == duplicate_count` always at Stage 2 and the summary takes
  the unchanged-wording branch (`src/main.rs:1531`). The divergent branch is
  unreachable from the binary. Confirmed.
- `scan_one` (`src/main.rs:1836`) flattens to the legacy tuple with
  `KeepPolicy::default()`, so all pre-existing unit tests are byte-identical.
- JSON gains two fields but they mirror the existing pair under neutral; verified
  in `tests/cli.rs:133-158`.

**One genuine behavior change at neutral** ‚ÄĒ the read-only listing **sort key**
(`src/main.rs:1502`). Stage 1 sorted groups by `group.iter().skip(1).sum(size)`;
Stage 2 sorts by `select_survivors(group, policy).victims.sum(size)`. Under
neutral these produce the **same multiset of bytes per group** (one survivor, so
victims = skip(1) by size since sizes are uniform), so the sort *key values* are
identical and order is preserved. I could not construct a divergence. So the claim
holds ‚ÄĒ but note it rests on "all sizes in a group are equal" (true today). If
that ever breaks, `skip(1)` (arbitrary element) vs `victims` (the non-survivor)
could pick different byte totals and reorder the listing. Documentary risk only.

**Conclusion:** neutral-invariance is real for Stage 2. No output or count
diverges from Stage 1 at the neutral default.

## 5. Test honesty ‚ÄĒ the metric tests are honest; the divergent summary string is untested [SHOULD-ADDRESS]

- **Could the new metric tests pass with a broken `select_survivors`?** No.
  `test_metrics_removable_less_than_redundancy` and
  `test_metrics_all_protected_zero_removable` (`src/main.rs:6340`-`6400`) create
  *real* files under real `00-a`/`00-b` dirs and run the full
  `find_duplicates_in_dirs` with a live `0*` policy, asserting removable ‚Č†
  redundancy. A `select_survivors` that ignored protect would fail them. Honest.
  `test_metrics_neutral_parity_on_mixed_tree` pins the neutral equivalence. Good.

- **But these tests use distinct physical files**, so collapse is a no-op in them
  ‚ÄĒ they do **not** cross-check collapse and metrics together. The collapse fix
  and the metric fix are tested in disjoint fixtures.

- **The divergent-summary *format string* is entirely untested.** The branch at
  `src/main.rs:1538-1547` (`"‚Ä¶ {} removable ({} reclaimable), {} protected and
  kept ‚Ä¶"`) is never reached by any test: the CLI is neutral (so integration tests
  hit the other branch) and the unit metric tests assert struct fields, not the
  formatted line. A transposed field, a wrong unit, or a mis-signed
  `duplicate_count - removable_files` in that format would ship undetected. A
  cheap fix: a unit test that formats the divergent summary from known metric
  values and asserts the string ‚ÄĒ or defer the string itself to Stage 3 where it
  first becomes reachable (the journal already deferred the `KEPT` labels for the
  same reason; the divergent summary arguably belongs with them).
