# Devil's Advocate — Stage 1: Selection Engine (`select_survivors`)

- **Date:** 2026-05-25
- **Branch:** `feature/configurable-survivor-selection`
- **Reviewer role:** Devil's advocate (read-only; no code changed)
- **Scope under challenge:** Stage 1 only — `select_survivors` + `KeepPolicy` /
  `FallbackStrategy` / `SurvivorReason`, wired behind `KeepPolicy::default()`
  (neutral). Two-metric accounting, explain output, protect-aware collapse,
  config, and CLI flags are explicitly later stages and are **not** challenged
  here except where Stage 1 leaves a latent trap for them.

## Verification performed

- `cargo build` — clean, no warnings.
- `cargo clippy` — clean, no lints.
- `cargo test` — 167 unit + 15 integration, all green. The 13 new
  `select_survivors` tests pass; `cli_read_only_lists_survivor_first` (the
  binary-level survivor-ordering invariant) still passes.

So "it compiles, it lints, the tests pass" is true. The question is what that
hides.

## 1. The "provably unchanged" claim — is it airtight?

**Verdict: airtight for the realistic production input, but the proof rests on a
precondition that is asserted nowhere in the code.**

The old behavior was `files.clone(); sort_by(survivor_cmp); skip(1)`. The new
default path is: partition (nothing protected, since both globsets are empty) →
`unprotected.sort_by(fallback_cmp(.., Lexical))` → `remove(0)` as survivor,
remainder as victims.

- `fallback_cmp(Lexical)` = `root.cmp().then(Ordering::Equal).then(path.cmp())`,
  which is *identically* `survivor_cmp` = `root.cmp().then(path.cmp())`. Same
  total order.
- Both sorts are `Vec::sort_by` (stable). Survivor = sorted[0]; victims =
  sorted[1..], in sorted order. So **victim order is preserved**, which is what
  matters for `get_unique_destination` collision renaming (`file(1).ext`,
  `file(2).ext`), action-log ordering, and partial-cancellation outcomes
  (cancellation breaks the victim loop mid-iteration — same loop, same order, so
  the same prefix of victims is acted on). Good. This is genuinely unchanged.

**The hidden precondition: distinct paths within a group.** Stability only
becomes irrelevant because the final `path.cmp()` tiebreak is *total* — but only
if no two entries in a group share a byte-identical path. If two equal paths
ever coexisted, `survivor_cmp`/`fallback_cmp` would return `Equal` for them, and
then:
- old code (stable sort) kept their relative input order;
- new code also uses a stable sort, so it *also* keeps their relative order —
  **so even this degenerate case is preserved.** I tried to construct a
  divergence here and could not: both paths run through `Vec::sort_by`, both
  stable, identical comparator. The claim survives even the nasty case.

In practice equal paths cannot reach `select_survivors` anyway:
`collapse_group_by_identity` runs before the `len > 1` filter and dedups by
`FileId` (dev+ino on Unix, path-string on Windows), and overlapping scan roots
are rejected with `OverlappingPaths`. So the precondition holds.

**Empty / singleton groups.** `process_duplicates` guards `files.len() <= 1`
and the read-only path filters `group.len() > 1`, so `select_survivors` is never
called with an empty group in production. **But the function itself is not
defensive:** with an empty `group` and nothing protected, `unprotected` is empty
and `victims.remove(0)` **panics** (index out of bounds). Today unreachable;
it becomes a live foot-gun the moment Stage 2's accounting or a future caller
invokes it on an arbitrary slice. A `debug_assert!(!group.is_empty())` or an
early return would cost nothing. **[SHOULD-ADDRESS]**

**Hash-map iteration order.** Unchanged — both old and new iterate
`for (hash, files) in duplicates` in the same `HashMap` order; the per-group
internal ordering is what changed, and it is preserved as shown above.

## 2. Determinism & total order of the fallback comparator

`fallback_cmp` = `root_index → strategy-key → path`. The trailing `path.cmp()`
makes the chain total whenever paths are distinct (always true post-collapse, see
§1). Ties at `root_index` and at the strategy key (equal mtime, equal component
count, `Lexical`'s constant `Equal`) all fall through to `path`. No non-total or
intransitive ordering in the pure-key strategies.

**One genuine soundness hazard — `Oldest`/`Newest` read the filesystem inside
the comparator.** `fallback_strategy_cmp` calls `mtime_of(a)` / `mtime_of(b)` on
*every comparison* during the sort. Consequences:

- **Repeated reads, not memoized.** An N-element group does O(N log N)
  `fs::metadata` syscalls instead of N. Duplicate groups are tiny so this is a
  perf footnote, not a bug.
- **TOCTOU / intermittent-failure non-determinism.** If a file's mtime changes
  mid-sort, or a metadata read succeeds on one comparison and fails on another
  (transient I/O error, file removed by another process), the comparator stops
  being a consistent total order *within a single sort*. `slice::sort_by`
  documents that an inconsistent comparator yields an unspecified-but-safe order
  (it will not panic or UB on stable Rust), so this won't crash — but it can pick
  a survivor that matches *neither* a clean "oldest" nor "newest" view, and the
  result is not reproducible run-to-run. The HLD §5.3 only promises determinism
  via the `path` tiebreak for the *unreadable* case; it does not address an mtime
  that is readable-but-changing or flaky. **The neutral default (`Lexical`) and
  `Shortest` never touch the filesystem, so Stage 1's shipped behavior is immune
  — this only bites once the CLI default (`oldest`) lands in a later stage.**
  Worth a one-line note that mtime should be read once per entry and cached
  before sorting, when that stage arrives. **[QUESTION]**

**Unreadable-mtime sentinel: direction is correct for BOTH directions.** Traced
by hand:
- `Oldest` = `cmp_oldest(key_a, key_b)` ascending; `None` is treated as
  `Greater` than any `Some`, so an unreadable file sorts *last* → loses the
  oldest contest. Correct.
- `Newest` = `cmp_oldest(key_b, key_a)` (arguments swapped = descending);
  `None` for `a` makes `cmp_oldest(Some_b, None_a) = Less`, so `a` sorts *first*
  → unreadable file wins newest. Correct, and matches the doc-comment's "treated
  as newest."

The `Option<SystemTime>` rewrite that replaced the Windows-overflowing "newest"
`SystemTime` sentinel is sound: `None` is given explicit `Greater`-than-`Some`
semantics in `cmp_oldest` rather than relying on `Option`'s derived ordering
(which would order `None < Some`, the wrong way). The implementer correctly did
**not** lean on derived `Option` ordering. Good catch on their part.

## 3. mtime correctness

- **Symlink vs target.** `mtime_of` uses `fs::metadata` (follows symlinks), so it
  reads the *target's* mtime, not the link's. For the dedup use-case that's
  arguably right (the content is the target's), and it is consistent with how the
  rest of the tool treats files. Not wrong, but undocumented — the HLD says
  "the path's modification time" without specifying link-following. **[QUESTION]**
- **Error paths.** `fs::metadata(...).and_then(modified()).ok()` collapses *both*
  a missing file and a platform that doesn't support `modified()` into `None`.
  HLD §5.3 says read failures "are surfaced on the error log when `error_logs` is
  set" — **`mtime_of` surfaces nothing; it silently swallows the error.** In
  Stage 1 nothing calls it with `error_logs` in scope, so this is deferred
  surface area, but the doc and the code already disagree. **[SHOULD-ADDRESS]**
  (later-stage concern, flagged so it isn't lost).

## 4. Test quality — proving behavior, or tautological?

Mostly good, with real gaps:

**Solid:**
- `test_select_survivors_neutral_default_matches_survivor_cmp` is the load-
  bearing one and it is *not* tautological: it sorts an independent copy with the
  real `survivor_cmp` and asserts both the survivor *and the full victim vector*
  match. This genuinely pins the "unchanged" invariant including victim order.
- `oldest`/`newest`/`shortest`/`lexical` tests each deliberately list the
  expected loser first, so a no-op or reversed implementation would fail. Not
  tautological.
- `equal_mtime_falls_to_path` exercises the `path` tiebreak with real equal
  mtimes.

**Gaps (a broken impl could pass the suite):**
1. **The unreadable-mtime sentinel branch is never exercised.** Every
   `Oldest`/`Newest` test sets `filetime` on files that *exist*. The `None` arm
   of `cmp_oldest` — the documented "treat as newest" sentinel, the exact thing
   that caused the Windows overflow rewrite — has **zero** coverage. Gut a branch
   (`None,Some => Less` instead of `Greater`) and the suite stays green. A test
   with one real file + one non-existent path under `Oldest` and under `Newest`
   would lock it. **[BLOCKING]** — this is the riskiest line in the diff and it's
   untested.
2. **No victim-order assertion when the *fallback* reorders with >2 victims.**
   `neutral_default` checks victim order, but the `oldest`/`shortest` tests have
   only a single victim, so they prove *who survives* but not that the remaining
   victims come out in chain order. Since victim order feeds move-collision
   renaming downstream, a 3-element fallback test asserting the full victim
   vector would be cheap insurance. **[SHOULD-ADDRESS]**
3. **Empty-group / singleton behavior of `select_survivors` is untested** (and
   would panic — see §1). **[SHOULD-ADDRESS]**
4. **`protect_by_dir` uses `Oldest` fallback but never reaches it** (a protected
   file exists, so the sort is skipped). Harmless, but it means "Oldest combined
   with a partially-protected group" is silently never run.
5. **Non-UTF-8 path components** (HLD §5.7 says these go through
   `to_string_lossy`) — untested. Low priority; rare and the path tiebreak still
   resolves. **[QUESTION]**

## 5. Scope discipline / hidden coupling

- **`overall_total` is still `Σ(len − 1)`** (`process_duplicates`, ~line 1216),
  computed independently of `select_survivors` rather than from `victims.len()`.
  Under the neutral default there is always exactly one survivor, so
  `len − 1 == victims.len()` and the `Successes: X / Y` denominator and JSON
  `total_candidates` are unchanged — **correct for Stage 1.** But it is a live
  trap for Stage 2: the moment protect retains >1 survivor, this denominator and
  the victim count diverge, and *both* must be migrated together (HLD §3.5/§5.5).
  It is not wired through `select_survivors` yet, so the "single source of truth"
  promise is only half-realized — the counters still use the old formula. Fine to
  defer, but it is the obvious place Stage 2 will introduce an off-by-one. Flag
  it loudly in the Stage 2 handoff. **[QUESTION]**
- **`build_globset` is `#[allow(dead_code)]`** — honest and correct (only tests
  call it in Stage 1). No leakage.
- **Dependencies pulled in early:** `globset` is a real (non-dev) dependency now
  even though Stage 1 ships only the neutral default that never compiles a
  non-empty glob in production. It's used by the new code paths and tests, so
  this is acceptable, not gold-plating — but note `toml` (HLD §6) is *not* yet
  added, consistent with config being deferred. Scope held.
- **`survivor_cmp` retained**, still used by `collapse_group_by_identity`. No
  dead-code warning. The HLD §5.6 protect-aware-collapse fix is correctly **not**
  attempted in Stage 1 (collapse still uses `survivor_cmp`, ignorant of policy) —
  which is the documented Stage-2 item, so no premature coupling. Good discipline.

---

## Bottom line

The neutral-default equivalence is genuinely airtight — I could not construct an
input where Stage 1 diverges from the old `sort_by(survivor_cmp) + skip(1)`,
including victim order, partial cancellation, equal-path degeneracy, and
empty/singleton groups (the last only because the callers guard `len > 1`). The
mtime sentinel direction is correct for both Oldest and Newest. The one real
defect is **test coverage**: the unreadable-mtime branch — the exact code that
forced the `Option<SystemTime>` rewrite — has no test and a broken version would
ship green. Add that test and a defensive guard on empty input, and Stage 1 is
clean.
