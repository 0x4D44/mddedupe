### SHOULD-FIX

**SF-1 — Protect-aware collapse safety property is effectively untested on Windows (the development
platform)**

`src/main.rs:6421` — `test_collapse_prefers_protected_alias` is `#[cfg(unix)]` and uses POSIX
hardlinks (`std::os::unix::fs::hard_link`) to give two paths the same `dev`+`ino` `FileId`. On
Windows the `FileId` implementation uses `path.canonicalize()` (`GetFinalPathNameByHandleW`), which
resolves NTFS junctions. Two paths pointing to the same physical file via a junction would share the
same canonical path and therefore the same `FileId::Path`, so the protect-aware selection logic in
`collapse_group_by_identity` would exercise the same branch. The logic is correct on Windows — it
just lacks a test.

The consequence: the most important safety property of Stage 2 (HLD §5.6, Risk §4 in HLD §10) has
zero test coverage on the platform where all development and CI runs. A regression in
`collapse_group_by_identity`'s protect branch on Windows would go undetected.

**Assessment of portability:** NTFS junction-based tests are straightforward in Rust using
`std::os::windows::fs::symlink_dir` (which creates a junction on Windows) or via a
`mklink /J` shell call. The `FileId::Path` path requires a junction (two directory entries for the
same physical directory; a file inside both paths would then have two entries in a scan), not a
hardlink to a file. This is more complex to set up than a Unix hardlink, but it is feasible.

**Recommended fix:** Add a second collapse-protect test gated `#[cfg(windows)]` (or
`#[cfg(not(unix))]`) that constructs a junction, puts a plain and a protected alias inside it,
assembles the group manually (matching the Unix test's approach), and asserts the protected
representative survives collapse. If junction creation is too brittle for a unit test, document
the gap explicitly in the test's doc comment and open a follow-up issue, rather than leaving it
silently untested.

**SF-2 — `process_duplicates` calls `select_survivors` twice per group (total + action loop)**

`src/main.rs:1297–1311` — `overall_total` is computed by iterating `duplicates.values()` and
calling `select_survivors(v, policy).1.len()` for every group. Then the action loop at line 1304
calls `select_survivors(files, policy)` again for each group. Both calls are pure (no mutation,
no I/O) and groups are tiny, so this is not a correctness problem or a performance problem in
practice. However, it is redundant work that makes the function harder to read: a reader seeing
the second call wonders whether it can ever produce a different result than the first.

Similarly, in the read-only display block (`run_app`, lines 1504–1515), `select_survivors` is
called once in `sort_by_key` and again in the display loop — two calls per group for display.

**Fix:** In `process_duplicates`, compute `overall_total` by accumulating victims during the
action loop after the first `select_survivors` call, then set `report.total_candidates` after the
loop. Alternatively, a single pre-pass that builds a `Vec<(victims, survivors)>` per group
eliminates both the double call and the `filter(len > 1)` duplication. The display path's
double call is lower priority but worth a comment noting that the two calls are intentionally
independent.

**SF-3 — `summary_detail` divergence condition may produce confusing output when
`removable_files > duplicate_count` (cannot happen today but is structurally unguarded)**

`src/main.rs:1532–1550` — The divergent branch prints `duplicate_count - removable_files` as
the "protected and kept" count. This subtraction is safe as long as
`removable_files <= duplicate_count`, which holds structurally (victims ≤ len−1 per group, so
`Σ victims ≤ Σ (len-1)`). However, there is no `debug_assert!` enforcing this invariant, and
the comment above the branch attributes the divergence only to "protect retains extras," which is
the only trigger today. A future code path that computes `removable_files` independently (e.g.
a pre-filter stage) could violate the invariant silently and produce a wrapping subtraction on
the `usize`.

**Fix:** Add `debug_assert!(removable_files <= duplicate_count, "removable cannot exceed
redundancy")` before the format string, or rewrite as
`duplicate_count.saturating_sub(removable_files)` with a comment. The saturating form also makes
the arithmetic intent explicit to future readers.

### NIT

**N-1 — `redundant_bytes` uses `skip(1)` on HashMap iteration order**

`src/main.rs:917` — `redundant_bytes` is computed as
`group.iter().skip(1).map(|(_, size, _)| *size).sum()`. This skips the first entry of whatever
order the HashMap chose for this group, not the survivor. This is correct because all files in a
hash group are byte-identical (same content → same size), so `skip(1)` on any permutation yields
`(len - 1) × size`. However, the code looks fragile to a reader who doesn't remember that the
size invariant holds across a hash group. A comment like
`// all files in a hash group have equal size; skip(1) is safe on any ordering` would prevent
a future "fix" that introduces an actual sort before the skip.

**N-2 — `DuplicateScan` is not `#[derive(Debug)]`**

`src/main.rs:658` — The struct has no `Debug` impl. It is an internal type used only inside
`find_duplicates_in_dirs` and destructured immediately in `run_app` and tests, so this is not a
functional gap. Add `#[derive(Debug)]` as a maintenance hygiene measure consistent with other
structs in the file.

**N-3 — `scan_one` doc comment says "neutral policy" but the policy origin is implicit**

`src/main.rs:1821–1825` — The doc comment accurately describes what `scan_one` does. However,
the comment "Tests that need the removable metrics or a non-neutral policy call
`find_duplicates_in_dirs` directly" is guidance for test authors, not a contract. This is fine
but worth a single-line note that `scan_one` always uses `KeepPolicy::default()` (neutral =
no protect rules, `Lexical` fallback) so callers do not need to look up the implementation.

**N-4 — `test_metrics_removable_less_than_redundancy` holds `lock_progress()` unnecessarily**

`src/main.rs:6450` — The `lock_progress` guard is acquired but this test does not touch env
vars or perform any action (it is read-only, `scan` only). The guard was added to three
`process_duplicates` tests that mutate/read env vars and perform deletes, which is correct. The
accounting tests do not need it. Holding the progress mutex delays other tests that genuinely
need it. Remove the guard from `test_metrics_removable_less_than_redundancy`,
`test_metrics_all_protected_zero_removable`, and `test_metrics_neutral_parity_on_mixed_tree`.

**N-5 — Divergent summary wording: "duplicate files" vs "duplicates found"**

`src/main.rs:1533` (equal branch) — "N duplicates found (M wasted)". Line 1542 (divergent
branch) — "N duplicate files (M)". The phrasing flips between "duplicates found" and "duplicate
files." Both are the `duplicate_count` (redundancy) metric. For consistency, align to one
phrasing. The equal branch is the common case that existing snapshot tests may rely on; the
divergent branch is the new one, so adjust it to match the equal branch.

**N-6 — `test_metrics_all_protected_zero_removable` asserts `redundant_bytes > 0` rather than
the exact value**

`src/main.rs:6497` — The test writes `b"same"` (4 bytes) to both files, so `redundant_bytes`
is exactly 4. A precise assertion `assert_eq!(scan.redundant_bytes, 4)` is stronger and
consistent with the `reclaimable_bytes == 0` assertion directly below it. The `> 0` form is not
wrong but is weaker than it needs to be.

**N-7 — Stage 2 comment in `run_app` is now stale**

`src/main.rs:1414–1416` — The comment reads "Stage 1: neutral selection policy (today's
behavior). Config/CLI resolution arrives in a later stage." This is accurate as of Stage 1 and
Stage 2 (policy resolution is still deferred), but the label "Stage 1:" is misleading now that
Stage 2 has shipped. Replace with "TODO(Stage 3): resolve policy from config/CLI" (the
`resolve_keep_policy` call per HLD §5.4).

## Accounting Correctness Audit

### Redundancy (`redundant_files`, `redundant_bytes`)

Formula: `Σ (group.len() - 1)` and `Σ (group.len() - 1) × size`. The `skip(1)` form for bytes
is equivalent because all entries in a hash group share the same size (same content → hashed from
the same size group → identical `size` field, see `find_duplicates_in_dirs` line 876 where `size`
comes from the outer size-group loop). Formula is correct and unchanged from Stage 1 semantics.

### Removable (`removable_files`, `reclaimable_bytes`)

Formula: `Σ victims.len()` and `Σ victim sizes` from `select_survivors(group, policy)`. Called
after `collapse_group_by_identity` (which may have reduced group size) and after the
`retain(len > 1)` filter (which removes any group that collapsed to a singleton). Both filters
run before the accounting loop, so the groups iterated are exactly the groups that will be acted
on. The formula is correct.

### `process_duplicates.overall_total`

Now set to `Σ select_survivors(v, policy).1.len()` over groups with `len > 1`. This matches the
accounting formula and the action loop's actual victim set. Consistent with `reclaimable_bytes`
semantics. The `filter(len > 1)` guard on line 1299 matches the `files.len() <= 1` continue on
line 1305, so no group is counted in `overall_total` but then skipped in the action loop (or
vice versa). Correct.

### `JsonSummary` fields

`removable_files` = `removable_files` from `DuplicateScan`; `reclaimable_bytes` =
`reclaimable_bytes` from `DuplicateScan`. Both are now serialized alongside the existing
`duplicate_files` / `duplicate_wasted_bytes`. The integration test verifies that under neutral
policy they equal the redundancy metrics. Wired correctly.

### Human summary "protected and kept" count

`duplicate_count - removable_files` = `Σ(len-1) - Σ victims` = number of non-victim extra
copies. Per the HLD §3.5 definition this is the count of files that are "extra copies above one
survivor" that protection shielded from removal. The formula matches the HLD example
("12 duplicate files, 9 removable, 3 protected and kept" → 12 - 9 = 3). Correct, subject to the
`debug_assert` recommendation in SF-3.

## Protect-Aware Collapse Correctness

The `(incoming_protected, current_protected)` match in `collapse_group_by_identity` has four
cases:

| incoming protected | current protected | `replace` | Correct? |
|---|---|---|---|
| true | false | true (promote to protected) | Yes — preferred alias must be protected |
| false | true | false (keep current protected) | Yes — already have a protected rep |
| true | true | `survivor_cmp`-first | Yes — both protected; arbitrary but deterministic |
| false | false | `survivor_cmp`-first | Yes — neutral fallback, preserves prior behavior |

The both-protected case (`true, true`) falls through to `survivor_cmp`, which is deterministic
and consistent with the neutral default. No alias is lost; the group is only collapsed when two
entries share a `FileId`.

The test `test_collapse_prefers_protected_alias` covers the `(false, true)` → don't replace
path: the `plain` alias (lower `root_index`, would normally win `survivor_cmp`) is encountered
first and registered. The `protected` alias arrives second as `incoming`, and because
`current_protected` is false and `incoming_protected` is true, `replace = true` promotes the
protected alias. The test correctly falsifies the case where `survivor_cmp` alone would make the
wrong choice. See SF-1 for the Windows coverage gap.

## Overall Verdict

Stage 2 is **sound and ready to commit** with two caveats that should be resolved first:

**SF-1** (Windows test gap for the critical collapse safety property) is the highest-priority
item. The logic is correct on both platforms, but correctness of the headline safety property
going untested on the development platform is an ongoing risk. At minimum, document the gap with
an explicit comment in `test_collapse_prefers_protected_alias` pointing to the Windows junction
approach.

**SF-2** (double `select_survivors` call in `process_duplicates`) is a code-quality concern, not
a bug. Acceptable to defer to Stage 3 if time is tight, but it creates a maintenance trap.

**SF-3** (`debug_assert` on the subtraction invariant) is a low-cost safety net. Worth adding
before the divergent summary branch ships.

NITs can be batched into Stage 3 or a separate hygiene commit.
