## 1. Did they kill the divergence, or move it?

**Killed, structurally.** Every consumer now reads the SAME `Vec<GroupPlan>`:

- `removable_files` / `reclaimable_bytes` — `run_app` derives both from
  `plans.iter()...` (`src/main.rs:1826`, `:1827`).
- Read-only listing — iterates `&plans` (`src/main.rs:1836`+).
- Action loop — `process_duplicates(&plans, ...)`; its signature changed from
  `&HashMap<..>` to `&[GroupPlan]` (`src/main.rs:1452`), so it physically cannot
  re-call `select_survivors`. The old `overall_total` re-counting pass is gone.
- JSON summary — reads the same `removable_files` / `reclaimable_bytes` locals
  (`src/main.rs:1991`–`1992`), identical to the text summary path. No second
  computation lurking in the JSON branch.

**Redundancy-vs-removable group membership — can they describe different groups?**
No. `find_duplicates_in_dirs` runs `collapse_group_by_identity` →
`retain(len>1)` → computes `redundant_files`/`redundant_bytes` over the FINAL map
(`src/main.rs:1043`–`1058`), then returns that map. `run_app` feeds the *same,
unmutated* map straight into `build_group_plans` (`src/main.rs:1822`). Both
metrics iterate identical group memberships; only the per-group *value* differs
(redundancy = `len-1`, policy-independent; removable = victim count under policy).
They can never reference different groups because the collapse happened once,
before either was derived. This is the right design.

**The only remaining count NOT from `plans`** is `redundant_files`/`_bytes`,
which is intentional and policy-independent. No leak.

## 2. Is the new divergence-agreement test load-bearing?

`test_group_plan_display_action_metrics_agree_under_oldest`
(`src/main.rs:7005`+) is load-bearing for ONE class of bug and silent on another:

- **Catches:** a broken `select_survivors`/`build_group_plans` that elects the
  lexical winner instead of the oldest. `assert_eq!(displayed_survivor, older)`
  with `oldest` deliberately disagreeing with `(root_index, path)` would fail.
  Good, genuine signal.
- **Does NOT exercise** the *original* race: two independent `select_survivors`
  calls disagreeing because a file's mtime changed between them. The test builds
  `plans` once and passes the same `plans` to `process_duplicates`, so divergence
  is impossible *by construction* — the test cannot reinject the two-call race.

This is fine because the anti-divergence guarantee now rests on the
`process_duplicates(&[GroupPlan])` **signature**, not on the test. But the
author's claim "verified it fails when divergence is reinjected" is only true for
the wrong-survivor class. **[QUESTION]** Worth being explicit in the journal: the
guarantee is enforced by the type (a revert to `&HashMap` is a compile error),
and the test guards survivor *identity*, not call-count. Don't oversell the test
as proof the race is gone — the type signature is the proof.

## 3. `build_group_plans` called once — really?

**Yes.** Single production call site: `src/main.rs:1822`. `process_duplicates`
takes the prebuilt slice and never rebuilds. `select_survivors` has no other
production caller. `select_survivors` reads each entry's mtime **exactly once**
into a presort key (`src/main.rs:255`–`265`); the comparator does zero I/O
(`fallback_cmp` doc + body, `:314`–`330`). No `O(n^2)`, no repeated mtime reads,
no borrow forcing a clone (plans hold `&` into the map, which lives to end of
`run_app`). Clean.

## 4. `--no-protect` ignoring config `fallback`

`resolve_keep_policy` (`src/main.rs:501`–`507`): under `--no-protect`, fallback =
`--keep` if given else `lexical`; the config `fallback` is dropped. Defensible —
"opt out of the policy" reasonably means "don't inherit config policy either," and
it's documented in the fn doc. The tests assert the *behavior*
(`test_resolve_keep_policy_no_protect_is_neutral_today_parity`,
`..._keep_overrides_neutral_fallback`).

**[QUESTION] — subtle, under-tested:** `--no-protect` ignores config *values* but
NOT config *validity*. `load_config` is called unconditionally at the top of
`resolve_keep_policy` (`src/main.rs:470`), before the `no_protect` branch. So
`mddedupe <dir> --no-protect` with a malformed/typo'd `./.mddedupe.toml` in CWD
still **fails fast and fatally**, even though `--no-protect` discards every value
that file would have supplied. That is arguably surprising ("I asked to ignore the
policy, why does my broken policy file still abort the run?"). No test pins this
either way. Decide intentionally: either skip `load_config` when `args.no_protect`
(and `--config` was not explicitly passed), or document that `--no-protect` still
validates a discovered config and add a test. Low severity, but it's a real
silent-vs-fatal edge nobody asserted.

## 5. Fail-fast action parse before scan

The action is parsed once, up front (`src/main.rs:1752`–`1780`), before the scan.
Net effect: unknown action / missing `--dest` / missing move destination now error
*before* the scan instead of after. **No run that previously succeeded now fails**
— the set of error conditions is unchanged; only their timing moved earlier, and
exit codes/`AppError` variants are identical (`cli_unknown_action_errors`,
`cli_move_requires_destination` still green). User-visible change: a failing
`--action bogus` no longer prints scan output first. Defensible improvement.

**[SHOULD-ADDRESS] dead-code duplication:** the `--create-dest` guard now exists
twice. `src/main.rs:1739`–`1743` rejects `create_dest` for *every* non-`move`
action (including `None`) and runs FIRST; the `None =>` arm at
`src/main.rs:1754`–`1756` repeats the `create_dest` check, but line 1739 has
already returned for the `None` case, so the inner check is unreachable. Harmless,
but it's dead code that invites "which one is authoritative?" confusion. Collapse
to one.

## 6. Cross-group ordering by reclaimable bytes — deterministic on ties?

`plans.sort_by_key(|p| p.victim_bytes())` (`src/main.rs:1825`) is stable, but its
input is `build_group_plans` iterating a `HashMap`, whose iteration order is
randomized per run. So **equal-victim-byte groups are emitted in nondeterministic
order**. HOWEVER this is **pre-existing** — the prior code did
`duplicates.iter().filter(..).collect()` then `sort_by_key` over the same
`HashMap` iterator, so ties were already run-to-run nondeterministic. The
restructure neither introduced nor worsened it. **[QUESTION]:** not a regression,
but if deterministic multi-group output is ever wanted (snapshot tests, diffable
logs), add `.then(hash)` to the sort key — cheap, and `hash` is already on the
plan. No current test asserts cross-group order, so nothing is flaky today.
Intra-group order IS fully deterministic (final `path` tiebreak, `:329`).

## Summary of findings

1. **[QUESTION]** The divergence-agreement test guards survivor *identity*, not the
   two-call race; the real anti-divergence guarantee is the
   `process_duplicates(&[GroupPlan])` signature. Don't credit the test with more
   than it proves; note the type-level guarantee in the journal.
2. **[QUESTION]** `--no-protect` ignores config *values* but still *loads and
   validates* a discovered `./.mddedupe.toml` — a broken config aborts a
   `--no-protect` run. Surprising and untested. Decide + document/test.
3. **[SHOULD-ADDRESS]** Dead-code: the `--create-dest` guard is duplicated; the
   copy inside the `None =>` action arm (`src/main.rs:1754`) is unreachable behind
   the earlier guard at `:1739`. Collapse to one.
4. **[QUESTION]** Cross-group ordering ties are nondeterministic (HashMap order) —
   pre-existing, not a regression; add `.then(plan.hash)` if deterministic output
   is ever desired.
5. **(clear)** Single-source plans, single `build_group_plans` call, single mtime
   read per entry, consistent JSON+text metrics, fail-fast timing-only behavior
   change — all verified sound.

**Verdict:** No BLOCKING issues — the restructure genuinely eliminates the
displayed-vs-deleted divergence at the type level; remaining items are clarity,
one dead-code line, and two intentional-decision points to pin down and test.
